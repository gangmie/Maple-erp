# Part 1. 공통 설계 원칙

## 1. 문서 목적

이 문서는 Maple ERP에서 사용하는 데이터베이스의 구조, 역할, 데이터 흐름, 명명 규칙을 정의합니다.

Maple ERP는 단순한 플레이 기록장이 아니라 다음 기능을 장기간 안정적으로 제공하는 개인 관리 시스템을 목표로 합니다.

- 캐릭터 정보 관리
- 플레이 활동 기록
- 보스 및 사냥 수익 계산
- 주간 진행 현황 관리
- 목표 메소 관리
- 메이플스토리 Open API 연동
- Notion API 자동 동기화
- GitHub Actions 기반 자동화
- 플레이 통계 및 Dashboard 제공

이 문서에서 정의한 데이터베이스 이름과 속성 이름은 이후 Notion API 코드에서도 사용합니다.

따라서 구현 이후에는 특별한 이유 없이 이름을 변경하지 않습니다.

---

# 2. 핵심 데이터 구조

Maple ERP의 데이터는 아래 네 개 계층으로 구분합니다.

1. 마스터 데이터
2. 활동 데이터
3. 거래 상세 데이터
4. 요약 데이터

```text
마스터 데이터
    │
    ├── 캐릭터
    ├── 보스 마스터
    └── 사냥터 마스터
          │
          ▼
활동 데이터
    │
    └── Activity
          │
          ├── 보스 기록
          ├── 사냥 기록
          └── 기타 수익 기록
                 │
                 ▼
요약 데이터
    │
    ├── 주간 관리
    └── 목표 메소
          │
          ▼
Dashboard
```

---

# 3. 각 데이터 계층의 역할

## 3.1 마스터 데이터

마스터 데이터는 여러 기록에서 반복해서 사용하는 기준 정보입니다.

예시:

- 캐릭터명
- 직업
- 월드
- 보스명
- 보스 난이도
- 결정석 기준 가격
- 사냥터명
- 지역
- 요구 포스

마스터 정보는 하나의 데이터베이스에서만 관리합니다.

보스 기록이나 사냥 기록에 동일한 정보를 반복해서 직접 입력하지 않고 Relation과 Rollup으로 가져옵니다.

---

## 3.2 활동 데이터

Activity는 사용자가 메이플스토리에서 수행한 모든 플레이 활동을 기록하는 공통 원장입니다.

예시:

- 보스 클리어
- 사냥
- 채집
- 제작
- 경매장 거래
- 몬스터파크
- 이벤트
- 무릉도장
- 기타 콘텐츠

Activity에는 모든 활동이 공통으로 가지는 정보만 저장합니다.

예시:

- 활동 날짜
- 캐릭터
- 활동 유형
- 소요 시간
- 완료 여부
- 주간 관리
- 메모

보스명, 사냥터, 결정석 수익처럼 특정 활동에만 필요한 정보는 Activity에 저장하지 않습니다.

---

## 3.3 거래 상세 데이터

거래 상세 데이터는 Activity에 연결되는 활동별 상세 기록입니다.

초기 구현 대상은 다음과 같습니다.

- 보스 기록
- 사냥 기록
- 기타 수익 기록

예를 들어 보스를 한 번 클리어했다면 다음 두 기록이 연결됩니다.

```text
Activity
- 날짜
- 캐릭터
- 활동 유형: 보스
- 소요 시간
- 완료 여부

보스 기록
- 보스
- 결정석 수익
- 드롭 아이템
- 예상 드롭 수익
- 확정 드롭 수익
```

Activity는 공통 정보의 원본이고, 보스 기록은 보스 전용 정보의 원본입니다.

---

## 3.4 요약 데이터

요약 데이터는 특정 기간이나 목표를 기준으로 활동 및 거래 데이터를 집계합니다.

예시:

- 캐릭터별 주간 보스 진행률
- 이번 주 결정석 수익
- 이번 주 사냥 수익
- 주간 총 플레이 시간
- 목표 메소 진행률

요약 데이터에는 가능하면 결과를 직접 입력하지 않습니다.

Relation, Rollup, Formula를 이용해 원본 데이터에서 계산합니다.

---

## 3.5 Dashboard

Dashboard는 데이터를 저장하는 데이터베이스가 아닙니다.

각 데이터베이스의 연결된 View와 집계 결과를 보여주는 화면입니다.

Dashboard에서는 원칙적으로 데이터를 직접 입력하지 않습니다.

다만 입력 편의를 위한 버튼이나 데이터베이스 템플릿은 배치할 수 있습니다.

예시:

- 사냥 기록 추가
- 보스 클리어 기록 추가
- 기타 수익 기록 추가
- 이번 주 관리 기록 생성

---

# 4. Activity 계층 설계

## 4.1 Activity의 목적

Activity는 모든 플레이 활동을 하나의 시간순 기록으로 관리하기 위한 공통 데이터베이스입니다.

Activity를 사용하면 다음 기능을 구현할 수 있습니다.

- 최근 활동 Timeline
- 오늘 수행한 활동 수
- 캐릭터별 플레이 시간
- 콘텐츠별 플레이 비중
- 주간 총 활동 수
- 월간 플레이 통계
- 마지막 플레이 날짜
- 활동 유형별 통합 검색

---

## 4.2 Activity와 상세 기록의 관계

Activity 한 건은 원칙적으로 하나의 상세 기록과 연결합니다.

```text
Activity 1건
    │
    ├── 보스 기록 0 또는 1건
    ├── 사냥 기록 0 또는 1건
    └── 기타 수익 기록 0 또는 1건
```

예시:

```text
Activity 유형이 "보스"인 경우
→ 보스 기록 1건과 연결

Activity 유형이 "사냥"인 경우
→ 사냥 기록 1건과 연결

Activity 유형이 "채집"인 경우
→ 기타 수익 기록 1건과 연결
```

하나의 Activity가 여러 종류의 상세 기록과 동시에 연결되지 않도록 합니다.

---

## 4.3 Activity가 필요한 이유

Activity 없이 보스 기록, 사냥 기록, 수익 기록만 각각 관리하면 다음과 같은 기능을 만들기 어렵습니다.

- 모든 활동을 시간순으로 한 번에 보기
- 전체 플레이 시간을 한 곳에서 계산하기
- 최근 플레이 활동을 통합해서 보여주기
- 콘텐츠 종류별 이용 비율 비교하기

Activity를 사용하면 상세 데이터베이스가 추가되더라도 공통 통계 구조를 변경할 필요가 없습니다.

예를 들어 추후 무릉도장 기록을 추가하더라도 다음처럼 확장할 수 있습니다.

```text
Activity
    ├── 보스 기록
    ├── 사냥 기록
    ├── 기타 수익 기록
    └── 무릉 기록
```

---

## 4.4 Activity의 한계와 구현 원칙

Notion Database에는 객체지향 프로그래밍의 상속 기능이 없습니다.

따라서 Activity와 상세 기록을 완전히 자동으로 하나의 객체처럼 처리할 수는 없습니다.

이를 보완하기 위해 다음 원칙을 적용합니다.

1. 공통 속성은 Activity에만 저장합니다.
2. 활동별 전용 속성은 상세 기록에만 저장합니다.
3. Activity와 상세 기록은 Relation으로 연결합니다.
4. 동일한 정보를 두 데이터베이스에 중복 저장하지 않습니다.
5. 템플릿 또는 버튼을 사용해 Activity와 상세 기록 생성을 간소화합니다.
6. 향후 Python 자동화로 두 기록의 생성을 동시에 처리할 수 있게 설계합니다.

# 6. 핵심 설계 원칙

## 6.1 Database First

모든 기능은 데이터베이스 구조를 먼저 설계한 뒤 구현합니다.

Dashboard나 버튼을 먼저 만들지 않습니다.

---

## 6.2 API First

메이플스토리 Open API에서 제공하는 데이터는 직접 입력하지 않는 것을 원칙으로 합니다.

API에서 제공하지 않는 정보만 사용자가 직접 입력합니다.

---

## 6.3 Single Source of Truth

하나의 정보는 하나의 원본 데이터베이스에서만 관리합니다.

예시:

- 캐릭터 직업의 원본: 캐릭터
- 결정석 기준 가격의 원본: 보스 마스터
- 보스 클리어 결과의 원본: 보스 기록
- 사냥 수익의 원본: 사냥 기록
- 활동 날짜와 소요 시간의 원본: Activity

다른 데이터베이스에서는 Relation, Rollup, Formula로 참조합니다.

---

## 6.4 Automation First

반복해서 입력하는 값은 템플릿, 버튼, Formula 또는 Python 자동화로 처리합니다.

자동화 후보:

- Activity 기록명 생성
- 주차 시작일 계산
- 주간 관리 기록 연결
- 보스 기록과 Activity 동시 생성
- 사냥 기록과 Activity 동시 생성
- 매주 새로운 주간 관리 기록 생성
- API 캐릭터 정보 갱신

---

## 6.5 데이터 누적 원칙

거래 데이터와 활동 데이터는 삭제보다 누적을 기본으로 합니다.

과거 기록을 수정해야 할 경우에도 가능한 한 기록을 삭제하지 않고 값을 수정합니다.

게임 업데이트로 기준값이 변경되어도 과거 기록은 당시 값을 유지해야 합니다.

---

## 6.6 확장성

다음 변화가 발생해도 기존 구조를 다시 만들지 않는 것을 목표로 합니다.

- 캐릭터 수 증가
- 새로운 월드 추가
- 새로운 보스 추가
- 새로운 사냥터 추가
- 새로운 콘텐츠 추가
- 다년간 데이터 누적
- Web Dashboard 추가
- Discord 알림 추가

# 8. 명명 규칙

## 8.1 문서 언어

- GitHub 문서: 한국어
- Notion 데이터베이스명: 한국어
- Notion 속성명: 한국어
- Python 파일명: 영어
- Python 변수명 및 함수명: 영어
- Git Commit Message: 한국어
- Git Commit 형식: [문서], [노션], [API], [자동화], [수정], [리팩토링], [릴리즈]
- 
---

## 8.2 데이터베이스 이름

데이터베이스 이름은 짧고 명확한 명사형으로 작성합니다.

사용 예시:

- 캐릭터
- 보스 마스터
- Activity
- 보스 기록
- 사냥 기록
- 주간 관리

사용하지 않는 예시:

- 캐릭터를 관리하는 데이터베이스
- 이번 주 보스 기록 목록
- 사냥해서 번 메소 기록

---

## 8.3 Notion 속성 이름

속성 이름은 사용자가 읽기 쉬운 한국어로 작성합니다.

가능하면 의미가 명확한 이름을 사용합니다.

예시:

- 최근 API 갱신 시간
- 예상 드롭 수익
- 확정 드롭 수익
- 결정석 일괄 판매 완료
- 활동 유형

모호한 축약어는 사용하지 않습니다.

---

## 8.4 날짜 속성

날짜 속성은 용도를 명확히 표시합니다.

예시:

- 활동 날짜
- 클리어 날짜
- 판매일
- 주차 시작일
- 주차 종료일
- 최근 API 갱신 시간

---

## 8.5 숫자 속성

숫자 속성에는 가능한 한 단위를 이름에 포함합니다.

예시:

- 소요 시간(분)
- 사냥 시간(분)
- 경험치 증가율(%)

메소 단위 숫자는 기본적으로 실제 메소 단위의 정수로 저장합니다.

예시:

```text
1억 메소 → 100000000
```

표시 형식은 Notion에서 천 단위 구분기호를 사용합니다.

|---|---|

| 에픽 던전 | 기타 수익 기록 |
| 무릉 | 기타 수익 기록 |
| 수로 | 기타 수익 기록 |
| 플래그 | 기타 수익 기록 |
| 기타 | 기타 수익 기록 |

Activity 하나에는 하나의 상세 기록만 연결됩니다.

하나의 상세 기록은 하나의 Activity에만 연결됩니다.

※ Notion Relation은 다대다 관계를 허용하지만, 본 프로젝트에서는 Formula와 Python 자동화를 통해 1:1 관계를 유지합니다.

## 9.3 필수 Relation

모든 Activity는 반드시 캐릭터와 연결되어야 합니다.

보스 기록, 사냥 기록, 기타 수익 기록은 반드시 Activity와 연결되어야 합니다.

주간 집계 대상 Activity는 주간 관리와 연결되어야 합니다.

## 9.4 과거 데이터 보존

- Dashboard 표시 여부
- 주간 관리 기록 생성 여부
- 주간 관리 자동 생성 여부

## 14.4 식별자 설계

캐릭터명은 사용자가 확인하기 쉬운 표시용 이름입니다.

캐릭터명은 변경될 수 있으므로 내부 식별자로 사용하지 않습니다.

Maple ERP에서는 다음 두 식별자를 구분합니다.

### 캐릭터 ID

Maple ERP 내부에서 사용하는 고유 식별자입니다.

예시:

```text
CHR-0001
CHR-0002
CHR-0003
```

캐릭터명이 변경되어도 캐릭터 ID는 변경하지 않습니다.

### OCID

Maple Open API에서 캐릭터 정보를 조회하기 위해 사용하는 외부 식별자입니다.

OCID는 API 동기화 과정에서 관리하며 캐릭터 ID와 별도로 저장합니다.

```text
캐릭터 ID
→ Maple ERP 내부 식별자

OCID
→ Maple Open API 조회용 외부 식별자
```

| 마지막 동기화 성공 시간 | 날짜 | 선택 | 자동화 | 예 | 마지막 정상 동기화 완료 시각 |
| API 기준 날짜 | 날짜 | 선택 | 자동화 | 예 | 조회한 게임 데이터의 기준 날짜 |

### 경험치 속성을 텍스트로 저장하는 이유

매우 큰 정수는 시스템 또는 연동 환경에 따라 숫자 정밀도 문제가 발생할 수 있습니다.

따라서 정확한 원본 경험치는 텍스트로 보관하고, Dashboard에서 활용하기 쉬운 경험치율은 별도 숫자 속성으로 저장합니다.

### 최근 API 갱신 시간과 마지막 동기화 성공 시간

```text
최근 API 갱신 시간
→ 성공 여부와 관계없이 마지막으로 동기화를 시도한 시각

마지막 동기화 성공 시간
→ 마지막으로 모든 필수 갱신이 정상 완료된 시각
```

동기화 실패가 발생하면 최근 API 갱신 시간은 변경될 수 있지만 마지막 동기화 성공 시간은 유지됩니다.

### 유니온 레벨 주의사항

유니온 정보는 캐릭터 자체의 고유 정보라기보다 계정과 월드 기준 정보에 가깝습니다.

초기 버전에서는 API 조회 편의를 위해 캐릭터에 저장합니다.

향후 동일한 유니온 값이 여러 캐릭터에 반복 저장되는 문제가 커지면 `계정·월드 프로필` 마스터로 분리합니다.

## 14.8 ERP 운영 설정 속성

| 속성명 | Notion 유형 | 기본값 | 입력 주체 | 설명 |
|---|---|---:|---|---|
| Dashboard 표시 | 체크박스 | 체크 | 사용자 | 메인 Dashboard 표시 여부 |
| 주간 관리 사용 | 체크박스 | 해제 | 사용자 | 해당 캐릭터의 주간 관리 기능 사용 여부 |
| 현재 관리 대상 | 체크박스 | 체크 | 사용자 | Maple ERP에서 현재 운영 및 관리하는 캐릭터인지 여부 |
| 관리 우선순위 | 선택 | 보통 | 사용자 | 높음, 보통, 낮음 |

### 설정별 역할

#### 현재 관리 대상

캐릭터가 현재 Maple ERP의 운영 대상인지 판단합니다.

해제할 경우 기본 자동화와 관리 View에서 제외합니다.

과거 Activity, 주간 관리 및 수익 기록은 유지합니다.

#### 주간 관리 사용

해당 캐릭터에게 주간 관리 기록이 필요한지 판단합니다.

현재 관리 대상이더라도 주간 콘텐츠를 관리하지 않는 캐릭터는 해제할 수 있습니다.

#### Dashboard 표시

Dashboard에 캐릭터를 보여줄지 결정합니다.

관리 대상이더라도 Dashboard 공간을 줄이기 위해 숨길 수 있습니다.

```text
현재 관리 대상 = 체크
Dashboard 표시 = 해제
```

조합도 허용합니다.

### 관리 우선순위 옵션

- 높음
- 보통
- 낮음

관리 우선순위는 Dashboard 정렬과 오늘의 플레이 브리핑에 사용합니다.

### 기본 목표 보스 수

```text
캐릭터의 기본 목표 보스 수
→ 새 주간 관리 기록의 목표 보스 수
```

향후 `캐릭터별 보스 설정` 데이터베이스가 구현되면 활성화된 목표 보스의 개수를 자동 계산하여 이 속성을 대체할 수 있습니다.

```text
캐릭터의 몬스터파크 주간 목표 횟수
→ 새 주간 관리 기록의 몬스터파크 목표 횟수
```

몬스터파크 관리를 사용하지 않는 캐릭터는 0으로 둘 수 있습니다.

- 대기
- 진행 중
- 성공
- 실패
- 비활성

### API 사용 여부와 자동 갱신 여부의 차이

`API 사용 여부`는 해당 캐릭터가 API 정보 관리 대상인지를 의미합니다.

`자동 갱신 여부`는 GitHub Actions가 정해진 시간에 자동으로 조회할지를 의미합니다.

예시:

```text
API 사용 여부 = 체크
자동 갱신 여부 = 해제
```

이 경우 수동 API 동기화는 가능하지만 정기 자동 갱신에서는 제외합니다.

### 주간 관리 사용과 자동 주간 생성의 차이

`주간 관리 사용`은 주간 관리 기능 자체를 사용하는지 나타냅니다.

`자동 주간 생성`은 Python 자동화가 매주 새 기록을 생성할지 나타냅니다.

예시:

```text
주간 관리 사용 = 체크
자동 주간 생성 = 해제
```

이 경우 사용자가 주간 관리 기록을 직접 생성하여 사용할 수 있습니다.


```text
현재 관리 대상이 해제되어 있으면

캐릭터 상태가 삭제 예정이면
"삭제 예정"

그 외에는
"관리 중"
```

```text

"정상"
```

## 14.17 자동화 규칙

### 신규 캐릭터 등록

1. 사용자가 캐릭터명을 입력합니다.
2. 자동화가 캐릭터 ID를 생성합니다.
3. 캐릭터명으로 OCID를 조회합니다.
4. 기본 정보를 가져옵니다.
5. API 속성을 갱신합니다.
6. 최근 API 갱신 시간을 기록합니다.
7. 마지막 동기화 성공 시간을 기록합니다.
8. 동기화 상태를 성공으로 변경합니다.
9. 연속 동기화 실패 횟수를 0으로 초기화합니다.
10. 사용자가 관리 설정과 주간 기본값을 확인합니다.

### 정기 API 동기화

자동화 대상 조건:

```text
현재 관리 대상 = 체크

API 사용 여부 = 체크

자동 갱신 여부 = 체크

캐릭터 상태가 활성 또는 육성 중
```

처리 순서:

1. 최근 API 갱신 시간 기록
2. 동기화 상태를 진행 중으로 변경
3. OCID 유효성 확인
4. 캐릭터 기본 정보 조회
5. 필요한 상세 API 조회
6. Notion API 속성 갱신
7. 성공 시 마지막 동기화 성공 시간 기록
8. 성공 시 동기화 상태를 성공으로 변경
9. 성공 시 연속 동기화 실패 횟수를 0으로 초기화
10. 실패 시 동기화 상태를 실패로 변경
11. 실패 시 오류 메시지 저장
12. 실패 시 연속 동기화 실패 횟수 1 증가

### 주간 관리 자동 생성

자동 생성 대상 조건:

```text
현재 관리 대상 = 체크

주간 관리 사용 = 체크

자동 주간 생성 = 체크

캐릭터 상태가 활성 또는 육성 중
```

생성 시 다음 값을 복사하거나 연결합니다.

```text
캐릭터
→ 주간 관리의 캐릭터

기본 목표 보스 수
→ 주간 관리의 목표 보스 수

몬스터파크 주간 목표 횟수
→ 주간 관리의 몬스터파크 목표 횟수

주간 기본 메모
→ 주간 관리의 주간 메모
```

기능별 관리 설정은 현재 버전에서 주간 관리의 Rollup을 통해 참조합니다.

향후 관리 설정 스냅샷이 구현되면 주간 기록 생성 시 기능별 관리 설정값도 일반 체크박스 속성으로 복사합니다.

### 닉네임 변경 대응

캐릭터 ID는 유지합니다.

API 조회 결과에서 캐릭터명이 변경된 경우 `캐릭터명`만 최신 이름으로 갱신합니다.

OCID 재조회가 필요한 경우 캐릭터명을 기준으로 다시 조회하고 조회 결과를 저장합니다.

## 14.20 데이터 무결성 규칙

1. 캐릭터 ID는 중복될 수 없습니다.
2. 캐릭터 ID는 생성 후 변경하지 않습니다.
3. 캐릭터명 변경 시 기존 캐릭터 행을 새로 생성하지 않습니다.
4. 캐릭터명이 변경되어도 기존 Activity 및 주간 관리 Relation을 유지합니다.
5. API가 갱신하는 속성과 사용자가 입력하는 관리 설정 속성을 구분합니다.
6. API 자동화는 사용자 관리 설정을 덮어쓰지 않습니다.
7. Activity 또는 수익 기록이 존재하는 캐릭터는 가능한 한 삭제하지 않습니다.
8. 사용하지 않는 캐릭터는 휴면 또는 은퇴 상태로 전환합니다.
9. 캐릭터 상태가 변경되어도 과거 기록은 유지합니다.
10. 현재 관리 대상이 해제된 캐릭터는 기본 자동화 대상에서 제외합니다.
11. API 사용 여부가 해제된 캐릭터는 API 동기화 대상에서 제외합니다.
12. 자동 갱신 여부는 API 사용 여부가 체크된 경우에만 사용할 수 있습니다.
13. 자동 주간 생성은 주간 관리 사용이 체크된 경우에만 사용할 수 있습니다.
14. 주간 관리 자동 생성 대상은 활성 또는 육성 중 캐릭터로 제한합니다.
15. 결정석 관리 사용 시 보스 관리도 활성화하는 것을 원칙으로 합니다.
16. 보스 관리 사용 시 기본 목표 보스 수를 설정해야 합니다.
17. 몬스터파크 관리 사용 시 몬스터파크 주간 목표 횟수를 설정해야 합니다.
18. 마지막 플레이 날짜는 Activity의 활동 날짜를 기준으로 계산합니다.
19. 유형별 누적 수익은 Activity 보조 Formula를 Rollup하여 계산합니다.
20. 보스 도전 수와 보스 클리어 수는 서로 구분합니다.
21. 결정석 수익과 전체 보스 수익은 서로 구분합니다.
22. 기능별 관리 설정 변경 시 현재 버전에서는 과거 주간 완료율이 변경될 수 있습니다.
23. 향후 관리 설정 스냅샷 도입 후에는 주간 기록 생성 당시 설정을 기준으로 완료율을 계산합니다.
24. 설정 경고가 정상이지 않은 캐릭터는 자동화 실행 전에 검토합니다.

## 15.2 목적

`보스 마스터`는 보스명과 난이도별 기준 정보를 한 곳에서 관리합니다.

보스 기록을 작성할 때 보스 이름, 난이도, 기준 결정석 가격 등을 매번 직접 입력하지 않고 Relation으로 선택할 수 있게 합니다.

보스명만으로는 난이도를 구분할 수 없으므로 한 행은 `보스명 + 난이도` 조합을 나타냅니다.

예시:

```text
스우 / Normal
스우 / Hard
스우 / Extreme
루시드 / Normal
루시드 / Hard
```

## 15.5 결정석 가격 관리

`결정석 기준 가격`에는 현재 기준 가격을 저장합니다.

실제 보스 클리어 기록에는 클리어 당시 결정석 가격을 별도 스냅샷으로 저장합니다.

따라서 보스 마스터의 가격을 변경해도 과거 보스 기록의 확정 수익은 바뀌지 않아야 합니다.

```text
보스 마스터
현재 결정석 기준 가격

↓

보스 기록 생성 시 복사

↓

실제 결정석 수익
```

결정석 가격 변경 이력이 필요해지면 향후 `결정석 가격 이력` 데이터베이스를 별도로 추가합니다.

- 보스 기록 생성 시 기본 예상 시간 복사

## 15.9 데이터 무결성 규칙

1. `보스명 + 난이도` 조합은 중복될 수 없습니다.
2. 보스 ID는 생성 후 변경하지 않습니다.
3. 결정석 가격 변경이 과거 기록에 반영되지 않도록 합니다.
4. 삭제된 콘텐츠도 과거 기록이 있으면 보스 마스터 행을 삭제하지 않습니다.
5. 사용하지 않는 보스는 `사용 여부`를 해제합니다.
6. 보스 기록에서는 반드시 보스 마스터를 Relation으로 선택합니다.
7. 보스별 결정석 판매 체크박스는 사용하지 않습니다.
8. 결정석 판매 완료 여부는 주간 관리에서 일괄 관리합니다.

## 16.5 최소 권장 레벨의 의미

`최소 권장 레벨`은 공식 고정값이 아니라 사용자가 실제 사냥 경험을 기준으로 정하는 값입니다.

직업, 장비, 전투력, 사냥 빌드에 따라 효율이 다를 수 있으므로 객관적인 입장 조건과 구분합니다.

`몬스터 레벨`은 사냥터 기준 정보이고, `최소 권장 레벨`은 개인 운영 정보입니다.

이 구조는 새로운 포스 체계가 추가되어도 속성을 다시 만들 필요가 없다는 장점이 있습니다.

2. 사냥터명 변경 시 기존 행을 새로 만들지 않습니다.
3. 사냥 기록은 반드시 사냥터 마스터와 연결합니다.
4. 현재 사용하지 않는 사냥터도 과거 기록이 있으면 삭제하지 않습니다.
5. 사용하지 않는 사냥터는 `사용 여부`를 해제합니다.
6. 사용자 평가와 객관적 기준 정보를 구분합니다.
7. 몬스터 레벨과 최소 권장 레벨을 같은 의미로 사용하지 않습니다.
8. 요구 포스는 포스 종류와 함께 입력합니다.
9. 게임 업데이트로 맵이 변경되면 최근 정보 확인일을 갱신합니다.

- 분당 처치 몬스터 수

- 경험치 효율 비교
- 버닝 수치 기록

- 사냥터 효율 추세 분석

# 17. 마스터 데이터 간 관계

```text
캐릭터
    │
    ├── Activity
    ├── 주간 관리
    └── 추천 사냥터
          │
          ▼
사냥터 마스터
    │
    └── 사냥 기록

보스 마스터
    │
    └── 보스 기록
```

초기 버전에서는 캐릭터, 보스 마스터, 사냥터 마스터가 서로 직접 강하게 의존하지 않도록 설계합니다.

실제 플레이 기록은 Activity와 거래 상세 데이터베이스를 통해 연결합니다.

# 18. 마스터 데이터 API 적용 범위

| 데이터베이스 | API 자동 갱신 | 사용자 직접 관리 |
|---|---|---|
| 캐릭터 | 기본 정보, 능력치, 심볼 등 | 운영 설정 및 기능 설정 |
| 보스 마스터 | 초기 버전에서는 없음 | 보스, 난이도, 결정석 기준 가격 |
| 사냥터 마스터 | 초기 버전에서는 없음 | 지역, 몬스터, 포스 및 개인 평가 |

API가 제공하는 캐릭터 데이터와 사용자가 설정하는 ERP 운영 속성을 명확히 구분합니다.

자동화는 사용자가 직접 설정한 관리 체크박스와 상태값을 수정하지 않습니다.

# 19. Part 2 확정 사항

이번 Part에서 확정한 사항은 다음과 같습니다.

- 캐릭터 마스터를 정보 저장소이자 ERP 설정 데이터베이스로 사용합니다.
- 캐릭터명과 OCID를 영구 내부 기본키로 사용하지 않습니다.
- Maple ERP 내부의 고정된 `캐릭터 ID`를 별도로 사용합니다.
- OCID는 Open API 조회를 위한 외부 식별자로 관리합니다.
- 캐릭터별 기능 관리 여부를 체크박스로 설정합니다.
- 캐릭터 상태를 이용해 과거 기록을 삭제하지 않고 운영 대상에서 제외합니다.
- API 정보와 사용자 설정 정보를 명확하게 분리합니다.
- 보스 마스터는 보스명과 난이도 조합을 한 행으로 관리합니다.
- 보스 마스터의 현재 결정석 가격과 과거 보스 기록의 실제 가격을 분리합니다.
- 결정석 판매 여부는 개별 보스 기록이 아니라 주간 관리에서 일괄 관리합니다.
- 사냥터 마스터는 객관적 정보와 사용자 개인 평가를 함께 관리합니다.
- 아케인포스와 어센틱포스 요구량을 `포스 종류 + 요구 포스` 구조로 통합합니다.
- 마스터 데이터는 삭제보다 비활성화를 기본으로 합니다.
- API가 제공하지 않는 보스 및 사냥터 기준 정보는 사용자가 직접 관리합니다.

# Part 3. 활동 및 거래 상세 데이터베이스 설계

# 20. 활동 및 거래 데이터 공통 원칙

Activity와 거래 상세 데이터베이스는 실제 플레이 결과가 발생할 때마다 기록이 누적되는 데이터베이스입니다.

초기 활동 및 거래 상세 데이터베이스는 다음과 같습니다.

| DB 코드 | 데이터베이스명 | 역할 |
|---|---|---|
| ACT | Activity | 모든 플레이 활동의 공통 원장 |
| BSR | 보스 기록 | 보스 클리어 및 수익 상세 |
| HTR | 사냥 기록 | 사냥 세션 및 효율 상세 |
| INR | 기타 수익 기록 | 채집, 제작, 경매장 등 기타 메소 변동 상세 |

---

## 20.1 공통 원칙

1. 활동 날짜, 캐릭터, 활동 유형, 소요 시간처럼 여러 활동에서 공통으로 사용하는 값은 Activity에 저장합니다.
2. 보스명, 사냥터, 결정석 수익처럼 특정 활동에서만 사용하는 값은 거래 상세 데이터베이스에 저장합니다.
3. Activity 한 건은 원칙적으로 하나의 거래 상세 기록과 연결합니다.
4. 하나의 Activity가 보스 기록과 사냥 기록에 동시에 연결되지 않도록 합니다.
5. 동일한 수익을 Activity와 거래 상세 데이터베이스에 각각 직접 입력하지 않습니다.
6. Activity의 예상 수익과 확정 수익은 거래 상세 데이터베이스에서 가져옵니다.
7. 거래 상세 기록이 존재하는 경우 연결된 Activity를 삭제하지 않습니다.
8. 과거 거래 기록은 삭제보다 상태 변경 또는 값 수정을 우선합니다.
9. Dashboard는 Activity와 거래 상세 데이터베이스의 View와 집계 결과만 표시합니다.
10. 향후 Python 자동화에서 Activity와 상세 기록을 동시에 생성할 수 있도록 설계합니다.

---

## 20.2 활동과 거래 상세의 관계

```text
Activity
    │
    ├── 보스 기록 0 또는 1건
    ├── 사냥 기록 0 또는 1건
    └── 기타 수익 기록 0 또는 1건
```

활동 유형과 연결 대상은 다음과 같습니다.

| 활동 유형 | 연결되는 상세 DB |
|---|---|
| 보스 | 보스 기록 |
| 사냥 | 사냥 기록 |
| 채집 | 기타 수익 기록 |
| 제작 | 기타 수익 기록 |
| 경매장 | 기타 수익 기록 |
| 이벤트 | 기타 수익 기록 |
| 몬스터파크 | 기타 수익 기록 또는 향후 전용 기록 |
| 무릉 | 향후 전용 기록 |
| 기타 | 기타 수익 기록 |

## 21.1 기본 정보

| 항목 | 내용 |
|---|---|
| 데이터베이스명 | Activity |
| DB 코드 | ACT |
| 데이터 분류 | 활동 데이터 |
| 핵심 역할 | 모든 플레이 활동의 공통 원장 |
| 제목 속성 | 활동명 |
| 주요 입력 주체 | 사용자, Notion 템플릿, Python 자동화 |
| 주요 연결 대상 | 캐릭터, 주간 관리, 보스 기록, 사냥 기록, 기타 수익 기록 |

---

## 21.2 목적

`Activity`는 메이플스토리에서 수행한 모든 플레이 활동을 시간순으로 관리하는 공통 데이터베이스입니다.

보스, 사냥, 채집, 제작처럼 성격이 다른 활동도 공통 속성은 동일하게 관리합니다.

각 Activity는 논리적으로 하나의 상세 기록과만 연결됩니다.

```text
보스 Activity
→ 보스 기록 1개

사냥 Activity
→ 사냥 기록 1개

기타 Activity
→ 기타 수익 기록 1개
```

Activity를 통해 다음 기능을 구현합니다.

- 최근 활동 Timeline
- 오늘 수행한 활동
- 캐릭터별 플레이 시간
- 콘텐츠별 플레이 비율
- 주간 활동 수
- 월간 활동 수
- 마지막 플레이 날짜
- 보스 도전 및 클리어 통계
- 활동 유형별 수익
- 활동 유형별 효율
- 주간 결정석 수익 집계
- Dashboard 최근 기록

## 21.3 Activity 기록 예시

### 보스 활동

```text
활동명: 2026-07-11 / 본캐 / 하드 윌
활동 날짜: 2026-07-11
캐릭터: 본캐
활동 유형: 보스
활동 상태: 완료
완료 여부: TRUE
소요 시간(분): 18
보스 기록: 하드 윌 클리어 기록
```

### 보스 실패 활동

```text
활동명: 2026-07-11 / 본캐 / 하드 윌 연습
활동 날짜: 2026-07-11
캐릭터: 본캐
활동 유형: 보스
활동 상태: 완료
완료 여부: TRUE
소요 시간(분): 12
보스 기록: 하드 윌 실패 기록
```

활동 자체가 끝났다면 `활동 상태`는 완료로 기록할 수 있습니다.

## 21.4 식별자 설계

### Activity ID

Maple ERP 내부 고유 식별자입니다.

예시:

```text
ACT-000001
ACT-000002
ACT-000003
```

Activity ID는 생성 이후 변경하지 않습니다.

### 활동명

Notion 제목 속성입니다.

권장 형식:

```text
날짜 / 캐릭터명 / 활동 내용
```

예시:

```text
2026-07-11 / 본캐 / 하드 윌
2026-07-11 / 본캐 / 도원경 사냥
2026-07-11 / 부캐1 / 채집
```

활동명은 사람이 읽기 위한 표시값이며 내부 식별자로 사용하지 않습니다.

## 21.5 속성

### 기본 속성

| 속성명 | Notion 유형 | 필수 | 입력 주체 | 설명 |
|---|---|---:|---|---|
| 활동명 | 제목 | 예 | 사용자/자동화 | 날짜, 캐릭터, 활동 내용을 조합한 제목 |
| Activity ID | 텍스트 | 예 | 자동화/사용자 | 내부 고유 식별자 |
| 활동 날짜 | 날짜 | 예 | 사용자/자동화 | 활동이 발생한 날짜와 시간 |
| 캐릭터 | Relation | 예 | 사용자/자동화 | 캐릭터 마스터 연결 |
| 활동 유형 | 선택 | 예 | 사용자/자동화 | 보스, 사냥, 채집 등 |
| 활동 상태 | 선택 | 예 | 사용자/자동화 | 예정, 진행 중, 완료, 취소 |
| 완료 여부 | Formula | 아니요 | 자동 계산 | 활동 상태가 완료이면 참 |
| 소요 시간(분) | 숫자 | 선택 | 사용자/자동화 | 실제 플레이 시간 |
| 주간 관리 | Relation | 선택 | 사용자/자동화 | 해당 캐릭터의 해당 주간 관리 기록 |
| 활동 당시 레벨 | 숫자 | 선택 | 자동화 | 활동 시점 캐릭터 레벨 스냅샷 |
| 등록 방식 | 선택 | 선택 | 사용자/자동화 | 수동, 템플릿, 버튼, API, 자동화, 가져오기 |
| 등록일 | 생성 시간 | 아니요 | Notion | 기록 생성 시각 |
| 수정일 | 최종 편집 시간 | 아니요 | Notion | 마지막 수정 시각 |
| 메모 | 텍스트 | 선택 | 사용자 | 자유 메모 |

### 상세 기록 Relation

| 속성명 | Notion 유형 | 필수 | 입력 주체 | 설명 |
|---|---|---:|---|---|
| 보스 기록 | Relation | 조건부 | 사용자/자동화 | 보스 활동의 상세 기록 |
| 사냥 기록 | Relation | 조건부 | 사용자/자동화 | 사냥 활동의 상세 기록 |
| 기타 수익 기록 | Relation | 조건부 | 사용자/자동화 | 기타 활동의 상세 기록 |

Activity 한 건에는 활동 유형에 맞는 상세 기록 하나만 연결합니다.

### 상세 기록 수익 Rollup

| 속성명 | Notion 유형 | 원본 속성 | 설명 |
|---|---|---|---|
| 보스 예상 수익 | Rollup | 보스 기록의 예상 총수익 | 보스 활동의 예상 수익 |
| 보스 확정 수익 | Rollup | 보스 기록의 확정 총수익 | 보스 활동의 확정 수익 |
| 사냥 예상 수익 | Rollup | 사냥 기록의 예상 총수익 | 사냥 활동의 예상 수익 |
| 사냥 확정 수익 | Rollup | 사냥 기록의 확정 총수익 | 사냥 활동의 확정 수익 |
| 기타 예상 수익 | Rollup | 기타 수익 기록의 예상 순수익 | 기타 활동의 예상 수익 |
| 기타 확정 수익 | Rollup | 기타 수익 기록의 확정 순수익 | 기타 활동의 확정 수익 |

### 보스 전용 Rollup

| 속성명 | Notion 유형 | 원본 속성 | 설명 |
|---|---|---|---|
| 보스 클리어 여부 | Rollup | 보스 기록의 클리어 여부 | 실제 보스 클리어 여부 |
| 보스 결정석 수익 | Rollup | 보스 기록의 개인 결정석 수익 | 개인에게 귀속되는 결정석 수익 |

### 계산 및 검증 속성

| 속성명 | Notion 유형 | 설명 |
|---|---|---|
| 예상 수익 | Formula | 보스, 사냥, 기타 예상 수익 합계 |
| 확정 수익 | Formula | 보스, 사냥, 기타 확정 수익 합계 |
| 시간당 수익 | Formula | 확정 수익을 소요 시간 기준으로 환산 |
| 상세 기록 연결 수 | Formula | 연결된 보스, 사냥, 기타 상세 기록의 총개수 |
| 연결 오류 상태 | Formula | 상세 기록 누락, 중복 연결, 유형 불일치 여부 |

## 21.7 활동 상태 옵션

초기 옵션은 다음과 같습니다.

- 예정
- 진행 중
- 완료
- 취소

`활동 상태`는 사용자가 직접 선택하는 속성입니다.

`완료 여부`는 활동 상태를 기준으로 자동 계산하는 Formula 속성입니다.

개념식:

```text
활동 상태가 "완료"이면 참

그렇지 않으면 거짓
```

보스 활동에서 `완료 여부`와 `보스 클리어 여부`는 서로 다른 의미입니다.

```text
완료 여부
→ Activity 기록의 수행 및 기록 절차가 끝났는지 판단

보스 클리어 여부
→ 보스전에서 실제로 보스를 클리어했는지 판단
```

예를 들어 보스 도전에 실패했더라도 도전이 종료되고 기록이 완료되었다면:

```text
활동 상태 = 완료
완료 여부 = TRUE
보스 클리어 여부 = FALSE
```

가 될 수 있습니다.

이 구조를 사용하는 이유는 다음과 같습니다.

- 활동 상태는 Activity의 현재 진행 단계를 표현합니다.
- 완료 여부는 필터, Rollup, 자동 집계에 사용합니다.
- 보스 클리어 여부는 실제 보스 성공 여부를 별도로 표현합니다.
- 사용자가 완료 여부를 직접 수정하지 않아 데이터 불일치를 방지합니다.

## 21.10 수익 Rollup 구조

Activity는 수익의 직접 입력 원본이 아닙니다.

상세 기록에서 유형별 예상 수익과 확정 수익을 Rollup한 뒤 Formula로 합산합니다.

### 보스 수익

```text
보스 기록의 예상 총수익
→ 보스 예상 수익 Rollup

보스 기록의 확정 총수익
→ 보스 확정 수익 Rollup
```

### 사냥 수익

```text
사냥 기록의 예상 총수익
→ 사냥 예상 수익 Rollup

사냥 기록의 확정 총수익
→ 사냥 확정 수익 Rollup
```

초기 버전에서 사냥 수익이 활동 종료 즉시 확정되는 구조라면 예상 수익과 확정 수익이 동일할 수 있습니다.

### 기타 수익

```text
기타 수익 기록의 예상 순수익
→ 기타 예상 수익 Rollup

기타 수익 기록의 확정 순수익
→ 기타 확정 수익 Rollup
```

### 결정석 수익

```text
보스 기록의 개인 결정석 수익
→ 보스 결정석 수익 Rollup
→ 주간 관리의 주간 결정석 총액으로 합산
```

`보스 결정석 수익`에는 드롭 아이템 수익을 포함하지 않습니다.

## 21.11 Formula

### 완료 여부

```text
활동 상태가 "완료"이면 참

그렇지 않으면 거짓
```

### 예상 수익

```text
보스 예상 수익
+ 사냥 예상 수익
+ 기타 예상 수익
```

### 확정 수익

```text
보스 확정 수익
+ 사냥 확정 수익
+ 기타 확정 수익
```

Activity 한 건은 하나의 상세 기록만 연결되므로 정상적인 경우 세 값 중 하나만 존재해야 합니다.

### 상세 기록 연결 수

개념식:

```text
보스 기록이 연결되어 있으면 1, 아니면 0

+ 사냥 기록이 연결되어 있으면 1, 아니면 0

+ 기타 수익 기록이 연결되어 있으면 1, 아니면 0
```

### 연결 오류 상태

개념식:

```text
상세 기록 연결 수가 0이면
"상세 기록 없음"

상세 기록 연결 수가 2 이상이면
"중복 연결"

활동 유형이 보스인데 보스 기록이 연결되지 않았으면
"유형 불일치"

활동 유형이 사냥인데 사냥 기록이 연결되지 않았으면
"유형 불일치"

활동 유형이 보스와 사냥이 아닌데
기타 수익 기록이 연결되지 않았으면
"유형 불일치"

그 외에는
"정상"
```

일부 활동은 수익 상세 기록이 필요하지 않을 수 있습니다.

수로, 플래그 등 별도 상세 DB가 아직 없는 활동은 초기 버전에서 `기타 수익 기록`을 선택적으로 연결하거나 Python 검증 대상에서 예외 처리합니다.

### 시간당 수익

```text
소요 시간(분)이 0보다 크면

확정 수익 ÷ 소요 시간(분) × 60

그렇지 않으면 0
```

## 21.12 자동화 규칙

### Activity 생성

1. Activity ID를 생성합니다.
2. 활동 날짜를 입력합니다.
3. 캐릭터를 연결합니다.
4. 활동 유형을 설정합니다.
5. 해당 주간 관리 기록을 찾습니다.
6. 활동 유형에 맞는 상세 기록을 생성합니다.
7. Activity와 상세 기록을 서로 연결합니다.
8. 활동 상태를 완료 또는 진행 중으로 설정합니다.
9. 완료 여부 Formula를 자동 계산합니다.
10. 상세 기록의 수익을 Activity에 Rollup합니다.
11. 연결 오류 상태를 확인합니다.

### 주간 관리 자동 연결

Activity 생성 시 다음 조건으로 주간 관리 기록을 찾습니다.

```text
캐릭터가 동일함

활동 날짜가 주차 시작일 이상

활동 날짜가 주차 종료일의 종료 시각 이하
```

활동 날짜에 시간 값이 포함될 수 있으므로 Python 자동화에서는 주차 종료일 당일의 모든 활동이 포함되도록 날짜 범위를 처리합니다.

해당 주간 관리 기록이 없다면 자동 생성 설정에 따라 새 기록을 생성할 수 있습니다.

### 활동 당시 레벨 스냅샷

Activity 생성 시 캐릭터 마스터의 현재 레벨을 복사합니다.

이 값은 이후 캐릭터 레벨이 상승해도 변경하지 않습니다.

### 연결 검증

Python 자동화는 다음 항목을 검사합니다.

```text
상세 기록 미연결
상세 기록 중복 연결
활동 유형과 상세 기록 유형 불일치
상세 기록의 역방향 Activity 중복 연결
캐릭터 불일치
활동 날짜 불일치
주간 관리 범위 불일치
```

## 21.14 데이터 무결성 규칙

1. Activity ID는 중복될 수 없습니다.
2. 모든 Activity는 하나의 캐릭터와 연결되어야 합니다.
3. Activity 한 건에는 활동 유형에 맞는 상세 기록 하나만 연결해야 합니다.
4. 보스 Activity에는 보스 기록만 연결합니다.
5. 사냥 Activity에는 사냥 기록만 연결합니다.
6. 초기 버전의 기타 활동은 기타 수익 기록과 연결할 수 있습니다.
7. 완료 여부는 사용자가 직접 입력하지 않고 활동 상태를 기준으로 계산합니다.
8. 완료 여부와 보스 클리어 여부는 서로 다른 의미입니다.
9. 보스 도전에 실패했더라도 Activity 자체는 완료 상태가 될 수 있습니다.
10. 완료된 Activity는 가능하면 삭제하지 않습니다.
11. 상세 기록 삭제 시 Activity 연결 오류 상태를 점검해야 합니다.
12. Activity의 소요 시간과 상세 기록의 소요 시간을 중복 저장하지 않습니다.
13. 활동 날짜는 Activity가 원본입니다.
14. 캐릭터는 Activity가 원본입니다.
15. 상세 기록에서는 Activity Relation을 통해 캐릭터와 날짜를 참조합니다.
16. 활동 당시 레벨은 스냅샷이므로 이후 자동 갱신하지 않습니다.
17. 예상 수익과 확정 수익은 상세 기록에서 가져오며 Activity에 직접 입력하지 않습니다.
18. 보스 결정석 수익에는 드롭 수익을 포함하지 않습니다.
19. 주간 관리에는 동일한 캐릭터와 해당 주차 범위의 Activity만 연결해야 합니다.
20. 연결 오류 상태가 정상이지 않은 Activity는 주간 및 Dashboard 집계 전에 검토해야 합니다.

## 22.2 목적

`보스 기록`은 한 번의 보스 도전 또는 클리어 결과를 기록합니다.

보스명과 기준 결정석 가격은 보스 마스터에서 가져오고, 실제 클리어 당시 수익은 스냅샷으로 저장합니다.

보스 수익은 다음 두 단계로 구분합니다.

- 예상 수익
- 확정 수익

결정석 수익은 클리어 시점에 사실상 확정되지만, 드롭 아이템은 실제 판매 전까지 예상 수익으로 취급합니다.

## 22.3 기록 단위

원칙적으로 한 번의 보스 클리어 또는 도전당 한 건을 생성합니다.

예시:

```text
2026-07-11 / 본캐 / 하드 윌
2026-07-11 / 부캐1 / 노말 스우
```

보스에 도전했으나 실패한 경우에도 기록할 수 있습니다.

## 22.6 도전 결과 옵션

초기 옵션은 다음과 같습니다.

- 클리어
- 실패
- 중단
- 연습
- 미진행

`도전 결과`는 사용자가 직접 선택하는 속성입니다.

`클리어 여부`는 Formula 속성으로, 도전 결과를 기준으로 자동 계산됩니다.

예시:

```text
도전 결과 = 클리어
클리어 여부 = TRUE
```

도전 결과가 클리어가 아니면 클리어 여부는 FALSE로 계산됩니다.

이 구조를 사용하는 이유는 다음과 같습니다.

- 도전 결과는 실제 플레이 결과를 표현합니다.
- 클리어 여부는 Formula를 이용하여 Rollup, 필터, 자동 집계 등에 활용합니다.
- 사용자가 직접 클리어 여부를 수정하지 않으므로 데이터 불일치가 발생하지 않습니다.

## 22.7 보스 보상 정산 방식

보스 보상은 결정석과 드롭 아이템으로 구분하여 정산합니다.

두 보상은 분배 방식이 다르므로 하나의 정산 비율을 공통으로 사용하지 않습니다.

### 결정석 정산

결정석 가격은 보스에 입장한 분배 인원을 기준으로 나누어 개인 수익을 계산합니다.

계산 기준:

```text
개인 결정석 수익
= 클리어 당시 결정석 가격 ÷ 결정석 분배 인원
```

예시:

| 결정석 분배 인원 | 개인 비율 |
|---:|---:|
| 1명 | 100% |
| 2명 | 50% |
| 3명 | 약 33.3333% |
| 4명 | 25% |
| 5명 | 20% |
| 6명 | 약 16.6667% |

결정석 분배 인원이 비어 있거나 0이면 계산 오류를 방지하기 위해 개인 결정석 수익은 0으로 계산합니다.

---

### 드롭 아이템 정산

결정석과 개인 보상을 제외한 파티 드롭 아이템은 다음 정산 방식 중 하나를 사용합니다.

- 솔로
- 균등 분배
- 블링크 독식
- 직접 입력

#### 솔로

혼자 클리어한 경우입니다.

```text
개인 드롭 정산 비율 = 100%
```

#### 균등 분배

파티 드롭의 가치를 인원수대로 동일하게 나눕니다.

```text
개인 드롭 정산 비율
= 100 ÷ 파티 인원
```

예시:

| 파티 인원 | 개인 비율 |
|---:|---:|
| 1명 | 100% |
| 2명 | 50% |
| 3명 | 약 33.3333% |
| 4명 | 25% |
| 5명 | 20% |
| 6명 | 약 16.6667% |

#### 블링크 독식

블링크 결과로 파티 보상을 한 명이 모두 가져가는 방식입니다.

```text
블링크 획득 여부 = 체크
→ 개인 드롭 정산 비율 = 100%

블링크 획득 여부 = 해제
→ 개인 드롭 정산 비율 = 0%
```

결정석, 솔 에르다의 기운 등 개인별 보상은 블링크 독식 대상에 포함하지 않습니다.

#### 직접 입력

고정 비율이 아니거나 별도의 합의 방식으로 정산하는 경우 사용합니다.

```text
개인 드롭 정산 비율
= 수동 드롭 정산 비율(%)
```

---

### 개인 드롭 수익 계산

```text
개인 예상 드롭 수익
= 예상 드롭 수익 × 개인 드롭 정산 비율(%)

개인 확정 드롭 수익
= 확정 드롭 수익 × 개인 드롭 정산 비율(%)
```

드롭 정산 방식이 비어 있으면 자동으로 계산하지 않거나, 솔로인 경우에만 100%를 적용합니다.

## 22.8 Formula

### 개인 결정석 수익

개념식:

```text
도전 결과가 "클리어"이고
결정석 분배 인원이 0보다 크면

클리어 당시 결정석 가격 ÷ 결정석 분배 인원

그렇지 않으면 0
```

### 개인 드롭 정산 비율

개념식:

```text
드롭 정산 방식이 "솔로"이면 100

드롭 정산 방식이 "균등 분배"이고
파티 인원이 0보다 크면
100 ÷ 파티 인원

드롭 정산 방식이 "블링크 독식"이고
블링크 획득 여부가 체크되어 있으면 100

드롭 정산 방식이 "블링크 독식"이고
블링크 획득 여부가 해제되어 있으면 0

드롭 정산 방식이 "직접 입력"이면
수동 드롭 정산 비율(%)

드롭 정산 방식이 비어 있거나
조건에 해당하지 않으면 0
```

### 개인 예상 드롭 수익

개념식:

```text
예상 드롭 수익
× 개인 드롭 정산 비율(%)
÷ 100
```

### 개인 확정 드롭 수익

개념식:

```text
확정 드롭 수익
× 개인 드롭 정산 비율(%)
÷ 100
```

## 22.9 결정석 판매 구조

결정석은 개별 보스 기록마다 판매 여부를 체크하지 않습니다.

클리어한 보스 기록의 결정석 금액은 주간 관리에서 합산합니다.

주간 관리에 다음 속성을 둡니다.

- 이번 주 결정석 총액
- 결정석 일괄 판매 완료
- 결정석 판매일

따라서 보스 기록에는 `결정석 판매 여부` 속성을 두지 않습니다.

## 22.10 드롭 수익 구조

공식 메이플스토리 Open API에서 메이플 옥션 실시간 시세를 직접 제공하지 않는 경우, 초기 버전에서는 사용자가 예상 시세와 실제 판매가를 입력합니다.

### 예상 드롭 수익

드롭 직후 또는 시세 확인 시 입력합니다.

### 확정 드롭 수익

실제 판매가 완료된 후 입력합니다.

### 드롭 판매 완료

판매가 끝나면 체크합니다.

### 실제 수익 반영 기준

Dashboard의 확정 수익에는 `확정 드롭 수익`만 반영합니다.

아직 판매되지 않은 드롭은 예상 수익에만 포함합니다.

## 22.15 데이터 무결성 규칙

1. 모든 보스 기록은 Activity와 연결되어야 합니다.
2. 연결된 Activity의 활동 유형은 보스여야 합니다.
3. 모든 보스 기록은 보스 마스터와 연결되어야 합니다.
4. 클리어 당시 결정석 가격은 생성 이후 자동으로 변경하지 않습니다.
5. 보스 마스터 가격 변경이 과거 기록에 영향을 주지 않습니다.
6. 예상 드롭 수익과 확정 드롭 수익을 구분합니다.
7. 드롭 판매 완료 시 확정 드롭 수익과 판매일을 입력합니다.
8. 결정석 판매 여부는 주간 관리에서 관리합니다.
9. 실패 기록에서도 소비 비용은 기록할 수 있습니다.
10. 파티 인원과 정산 비율의 합리성을 사용자가 확인합니다.
11. 파티 인원과 결정석 분배 인원은 1 이상이어야 합니다.
12. 결정석 수익과 드롭 수익은 서로 다른 분배 규칙을 적용합니다.
13. 균등 분배에서는 개인 드롭 정산 비율을 `100 ÷ 파티 인원`으로 자동 계산합니다.
14. 블링크 독식에서는 획득자는 100%, 미획득자는 0%로 계산합니다.
15. 직접 입력 방식의 정산 비율은 0% 이상 100% 이하로 입력합니다.
16. 솔 에르다의 기운 등 개인 보상은 파티 드롭 수익에 포함하지 않습니다.
17. 블링크 독식은 파티 드롭 아이템에만 적용하며, 결정석 및 개인 보상에는 적용하지 않습니다.

## 23.2 목적

`사냥 기록`은 한 번의 연속된 사냥 세션을 기준으로 메소, 경험치, 아이템 판매 및 사냥 효율을 기록합니다.

사냥 날짜, 캐릭터, 활동 당시 레벨, 소요 시간은 Activity에서 관리합니다.

사냥 기록에는 다음과 같은 사냥 전용 정보만 저장합니다.

- 사냥터
- 메소 측정 방식
- 시작 및 종료 메소
- 직접 획득 메소
- 소비 메소
- 판매 아이템 예상 수익
- 판매 아이템 확정 수익
- 경험치 변화
- 사냥 버프
- 메소 획득량 및 아이템 드롭률
- 버닝 단계
- 처치 마릿수

```text
Activity
→ 날짜, 캐릭터, 활동 당시 레벨, 사냥 시간

사냥 기록
→ 사냥터, 메소, 경험치, 버프, 아이템 및 효율
```

## 23.3 기록 단위

한 번의 연속된 사냥 세션마다 사냥 기록 한 건을 생성합니다.

예시:

```text
2026-07-11 / 본캐 / 도원경
2026-07-12 / 본캐 / 아르테리아
```

다음 상황에서는 별도 기록으로 분리하는 것을 권장합니다.

- 사냥터를 변경한 경우
- 캐릭터를 변경한 경우
- 측정 방식을 변경한 경우
- 장시간 휴식 후 다시 시작한 경우
- 사냥 버프 구성이 크게 달라진 경우
- 메소 또는 경험치 측정 기준을 새로 시작한 경우

짧은 포탈 이동이나 일시적인 중단은 동일한 세션으로 유지할 수 있습니다.

---

## 23.4 식별자 설계

### 사냥 기록 ID

Maple ERP 내부 고유 식별자입니다.

예시:

```text
HTR-000001
HTR-000002
HTR-000003
```

사냥 기록 ID는 생성 이후 변경하지 않습니다.

### 기록명

Notion 제목 속성입니다.

권장 형식:

```text
날짜 / 캐릭터명 / 사냥터명
```

예시:

```text
2026-07-11 / 본캐 / 도원경
2026-07-12 / 본캐 / 아르테리아
```

기록명은 사람이 읽기 위한 표시값이며 내부 식별자로 사용하지 않습니다.

## 23.7 메소 측정 속성

| 속성명 | Notion 유형 | 필수 | 입력 주체 | 설명 |
|---|---|---:|---|---|
| 메소 측정 방식 | 선택 | 예 | 사용자 | 시작·종료 메소 또는 직접 획득 메소 |
| 시작 메소 | 숫자 | 조건부 | 사용자 | 사냥 시작 전 보유 메소 |
| 종료 메소 | 숫자 | 조건부 | 사용자 | 사냥 종료 후 보유 메소 |
| 직접 획득 메소 | 숫자 | 조건부 | 사용자 | 게임 측정값 또는 별도 계산 결과 |
| 소비 메소 | 숫자 | 선택 | 사용자 | 사냥 세션에서 사용한 비용 |
| 순수 메소 수익 | Formula | 아니요 | 자동 계산 | 측정 방식에 따라 계산된 순수 메소 수익 |
| 메소 입력 상태 | Formula | 아니요 | 자동 계산 | 측정값 누락 또는 중복 입력 검증 |

### 메소 측정 방식 옵션

- 시작·종료 메소
- 직접 획득 메소

### 시작·종료 메소 방식

사냥 전후의 보유 메소 차이를 기준으로 수익을 계산합니다.

```text
종료 메소
- 시작 메소
- 소비 메소
```

이 방식은 입력이 간단하지만 다음 수입과 지출이 사냥 중 섞이면 오차가 발생할 수 있습니다.

- 경매장 정산
- 아이템 구매
- 보스 결정석 판매
- 다른 캐릭터로 메소 이동
- 강화 비용 지출
- 기타 메소 변동

### 직접 획득 메소 방식

게임 측정값이나 별도 측정 결과를 직접 입력합니다.

```text
직접 획득 메소
- 소비 메소
```

메소 측정 방식을 선택 속성으로 분리하여 두 방식이 동시에 입력되었을 때 발생할 수 있는 계산 모호성을 방지합니다.

## 23.8 아이템 판매 속성

| 속성명 | Notion 유형 | 필수 | 입력 주체 | 설명 |
|---|---|---:|---|---|
| 판매 아이템 예상 수익 | 숫자 | 선택 | 사용자 | 미판매 아이템의 현재 예상 판매가 |
| 판매 아이템 확정 수익 | 숫자 | 선택 | 사용자 | 실제 판매가 완료된 금액 |
| 아이템 판매 완료 | 체크박스 | 선택 | 사용자 | 사냥 드롭 아이템 판매 완료 여부 |
| 아이템 판매일 | 날짜 | 선택 | 사용자 | 실제 판매가 완료된 날짜 |
| 예상 총수익 | Formula | 아니요 | 자동 계산 | 순수 메소 수익과 예상 판매 수익의 합계 |
| 확정 총수익 | Formula | 아니요 | 자동 계산 | 순수 메소 수익과 확정 판매 수익의 합계 |
| 미확정 예상 수익 | Formula | 아니요 | 자동 계산 | 아직 확정되지 않은 예상 아이템 수익 |
| 판매 정산 상태 | Formula | 아니요 | 자동 계산 | 판매 완료 설정과 금액 입력 상태 검증 |

### 예상 수익과 확정 수익

```text
판매 아이템 예상 수익
→ 아직 판매되지 않은 아이템의 예상 가치

판매 아이템 확정 수익
→ 실제 판매가 끝난 아이템의 확정 금액
```

아이템 판매가 완료되지 않았다면 예상 수익에만 포함합니다.

실제 판매가 끝난 뒤에는 확정 판매 금액과 판매일을 입력합니다.

## 23.9 경험치 측정 속성

| 속성명 | Notion 유형 | 필수 | 입력 주체 | 설명 |
|---|---|---:|---|---|
| 시작 경험치율(%) | 숫자 | 선택 | 사용자/자동화 | 사냥 시작 전 경험치 진행률 |
| 종료 경험치율(%) | 숫자 | 선택 | 사용자 | 사냥 종료 후 경험치 진행률 |
| 종료 레벨 | 숫자 | 선택 | 사용자/자동화 | 사냥 종료 시 캐릭터 레벨 |
| 직접 경험치 증가율(%) | 숫자 | 선택 | 사용자 | 레벨업 발생 등 자동 계산이 어려울 때 입력 |
| 경험치 증가율(%) | Formula | 아니요 | 자동 계산 | 직접 입력값 또는 시작·종료 경험치율 기준 증가량 |
| 시간당 경험치율(%) | Formula | 아니요 | 자동 계산 | 시간당 경험치 증가율 |
| 경험치 입력 상태 | Formula | 아니요 | 자동 계산 | 레벨업 및 경험치 입력 상태 검증 |

시작 레벨은 사냥 기록에 직접 입력하지 않습니다.

Activity의 `활동 당시 레벨`을 Rollup하여 사용합니다.

### 직접 경험치 증가율

레벨업이 발생하지 않았다면 다음 계산을 사용할 수 있습니다.

```text
종료 경험치율(%)
- 시작 경험치율(%)
```

레벨업이 발생하면 단순 차감으로 정확한 경험치 증가량을 계산하기 어렵습니다.

예시:

```text
시작 레벨: 280
시작 경험치율: 95%

종료 레벨: 281
종료 경험치율: 10%
```

단순 계산:

```text
10 - 95 = -85
```

이 값은 실제 경험치 증가량을 의미하지 않습니다.

따라서 레벨업이 발생했거나 자동 계산이 부정확한 경우 `직접 경험치 증가율(%)`을 입력합니다.

Formula는 직접 입력값을 우선 사용합니다.

| MVP 사용 여부 | 체크박스 | 선택 | 사용자 | MVP 추가 경험치 버프 사용 여부 |

따라서 이후 캐릭터 장비나 스펙이 변경되어도 기존 사냥 기록은 변경하지 않습니다.

## 23.11 Rollup 속성

| 속성명 | Relation | 원본 속성 | 계산 방식 | 설명 |
|---|---|---|---|---|
| 활동 날짜 | Activity | 활동 날짜 | 원본 표시 | 사냥 날짜 |
| 캐릭터 | Activity | 캐릭터 | 원본 표시 | 플레이 캐릭터 |
| 사냥 시간(분) | Activity | 소요 시간(분) | 원본 표시 | 실제 사냥 시간 |
| 시작 레벨 | Activity | 활동 당시 레벨 | 원본 표시 | 사냥 시작 시점의 캐릭터 레벨 |
| 활동 상태 | Activity | 활동 상태 | 원본 표시 | 사냥 Activity의 진행 상태 |
| 완료 여부 | Activity | 완료 여부 | 원본 표시 | 사냥 세션 완료 여부 |
| 사냥터명 | 사냥터 | 사냥터명 | 원본 표시 | 사냥터 이름 |
| 지역 | 사냥터 | 지역 | 원본 표시 | 사냥터 지역 |
| 몬스터 레벨 | 사냥터 | 몬스터 레벨 | 원본 표시 | 주요 몬스터 레벨 |
| 포스 종류 | 사냥터 | 포스 종류 | 원본 표시 | 요구 포스 종류 |
| 요구 포스 | 사냥터 | 요구 포스 | 원본 표시 | 사냥터 요구 포스 |

`시작 레벨`은 사용자가 직접 입력하지 않습니다.

```text
캐릭터의 현재 레벨
→ Activity 생성 시 활동 당시 레벨에 복사
→ 사냥 기록에서 시작 레벨로 Rollup
```

이 구조를 통해 같은 값을 Activity와 사냥 기록에 중복 입력하지 않습니다.

## 23.12 Formula

### 순수 메소 수익

개념식:

```text
메소 측정 방식이 "직접 획득 메소"이고
직접 획득 메소가 입력되어 있으면

직접 획득 메소
- 소비 메소

메소 측정 방식이 "시작·종료 메소"이고
시작 메소와 종료 메소가 모두 입력되어 있으면

종료 메소
- 시작 메소
- 소비 메소

그렇지 않으면 0
```

순수 메소 수익은 음수가 될 수 있습니다.

사냥 중 소비한 비용이 획득 메소보다 많다면 음수로 표시합니다.

### 예상 총수익

```text
순수 메소 수익
+ 판매 아이템 예상 수익
```

### 확정 총수익

```text
순수 메소 수익
+ 판매 아이템 확정 수익
```

Activity의 사냥 수익 Rollup은 다음 속성을 사용합니다.

```text
사냥 기록의 예상 총수익
→ Activity의 사냥 예상 수익

사냥 기록의 확정 총수익
→ Activity의 사냥 확정 수익
```

### 미확정 예상 수익

```text
판매 아이템 예상 수익
- 판매 아이템 확정 수익이 0보다 크면

판매 아이템 예상 수익
- 판매 아이템 확정 수익

그렇지 않으면 0
```

미확정 예상 수익은 0 미만으로 계산하지 않습니다.

### 시간당 메소

```text
사냥 시간(분)이 0보다 크면

확정 총수익
÷ 사냥 시간(분)
× 60

그렇지 않으면 0
```

시간당 메소는 확정 총수익을 기준으로 계산합니다.

미판매 아이템 예상 수익은 시간당 확정 메소에 포함하지 않습니다.

### 경험치 증가율(%)

```text
직접 경험치 증가율(%)이 입력되어 있으면

직접 경험치 증가율(%)

시작 레벨과 종료 레벨이 같고
시작 경험치율(%)과 종료 경험치율(%)이 입력되어 있으면

종료 경험치율(%)
- 시작 경험치율(%)

그렇지 않으면 0
```

레벨업이 발생했는데 직접 경험치 증가율이 입력되지 않은 경우 자동 계산하지 않습니다.

### 시간당 경험치율(%)

```text
사냥 시간(분)이 0보다 크면

경험치 증가율(%)
÷ 사냥 시간(분)
× 60

그렇지 않으면 0
```

### 시간당 마릿수

```text
사냥 시간(분)이 0보다 크면

마릿수
÷ 사냥 시간(분)
× 60

그렇지 않으면 0
```

## 23.13 검증 Formula

### 메소 입력 상태

개념식:

```text
메소 측정 방식이 비어 있으면
"측정 방식 미설정"

메소 측정 방식이 "시작·종료 메소"이고
시작 메소 또는 종료 메소가 비어 있으면
"시작·종료 메소 미입력"

메소 측정 방식이 "직접 획득 메소"이고
직접 획득 메소가 비어 있으면
"직접 획득 메소 미입력"

메소 측정 방식이 "시작·종료 메소"인데
직접 획득 메소도 입력되어 있으면
"불필요한 직접 획득 메소 입력"

메소 측정 방식이 "직접 획득 메소"인데
시작 메소 또는 종료 메소가 입력되어 있으면
"불필요한 시작·종료 메소 입력"

그 외에는
"정상"
```

### 판매 정산 상태

개념식:

```text
아이템 판매 완료가 체크되어 있는데
판매 아이템 확정 수익이 비어 있으면
"확정 수익 미입력"

아이템 판매 완료가 체크되어 있는데
아이템 판매일이 비어 있으면
"판매일 미입력"

아이템 판매 완료가 해제되어 있는데
판매 아이템 확정 수익이 입력되어 있으면
"판매 완료 확인 필요"

그 외에는
"정상"
```

판매 아이템이 없는 세션은 예상 수익, 확정 수익 및 판매일을 모두 비워둘 수 있습니다.

### 경험치 입력 상태

개념식:

```text
시작 레벨과 종료 레벨이 다르고
직접 경험치 증가율(%)이 비어 있으면
"레벨업 경험치 보정 필요"

시작 경험치율(%)만 입력되어 있거나
종료 경험치율(%)만 입력되어 있으면
"경험치율 입력 누락"

직접 경험치 증가율(%)이 0보다 작으면
"경험치 증가율 확인 필요"

그 외에는
"정상"
```

### 연결 상태

개념식:

```text
Activity가 연결되지 않았으면
"Activity 미연결"

Activity가 둘 이상 연결되어 있으면
"Activity 중복 연결"

연결된 Activity의 활동 유형이 사냥이 아니면
"활동 유형 불일치"

사냥터가 연결되지 않았으면
"사냥터 미연결"

그 외에는
"정상"
```

Notion Formula만으로 Relation 개수 또는 Rollup 값을 정확하게 검증하기 어려운 부분은 Python 자동화에서도 확인합니다.

## 23.14 Relation

| 속성명 | 연결 DB | 논리적 관계 | 설명 |
|---|---|---|---|
| Activity | Activity | 1:1 | 공통 활동 기록 |
| 사냥터 | 사냥터 마스터 | N:1 | 사냥터 기준 정보 |

Notion Relation은 물리적으로 여러 페이지를 연결할 수 있습니다.

그러나 Maple ERP에서는 사냥 기록 한 건에 Activity 한 건만 연결하는 논리적 1:1 규칙을 적용합니다.

## 23.15 자동화 규칙

### 신규 사냥 기록 생성

1. Activity를 생성합니다.
2. Activity ID를 생성합니다.
3. 활동 유형을 `사냥`으로 설정합니다.
4. 활동 상태를 `진행 중` 또는 `완료`로 설정합니다.
5. 캐릭터를 연결합니다.
6. 캐릭터의 현재 레벨을 Activity의 `활동 당시 레벨`에 복사합니다.
7. 해당 주간 관리 기록을 찾아 연결합니다.
8. 사냥 기록을 생성합니다.
9. 사냥 기록 ID를 생성합니다.
10. Activity와 사냥 기록을 서로 연결합니다.
11. 사냥터를 연결합니다.
12. 메소 측정 방식을 설정합니다.
13. 기록명을 자동 구성합니다.
14. 연결 상태를 검증합니다.

### 사냥 시작 처리

3. 소비 메소를 입력합니다.
4. 종료 경험치율을 입력합니다.
5. 종료 레벨을 입력합니다.
6. 필요하면 직접 경험치 증가율을 입력합니다.
7. 마릿수를 입력합니다.
8. Activity 상태를 완료로 변경합니다.
9. 완료 여부 Formula를 갱신합니다.
10. 예상 총수익과 확정 총수익을 계산합니다.
11. 메소 입력 상태를 검증합니다.
12. 경험치 입력 상태를 검증합니다.
13. Activity에 수익을 Rollup합니다.

2. `아이템 판매 완료`를 체크합니다.
3. `아이템 판매일`을 입력합니다.
4. 판매 정산 상태를 확인합니다.
5. 사냥 기록의 확정 총수익을 갱신합니다.
6. Activity의 사냥 확정 수익을 갱신합니다.
7. 주간 관리와 캐릭터의 수익 Rollup을 갱신합니다.

## 23.18 데이터 무결성 규칙

1. 사냥 기록 ID는 중복될 수 없습니다.
2. 모든 사냥 기록은 Activity 한 건과 연결되어야 합니다.
3. 연결된 Activity의 활동 유형은 반드시 사냥이어야 합니다.
4. 사냥 기록 한 건에 Activity를 둘 이상 연결하지 않습니다.
5. 모든 사냥 기록은 사냥터 마스터와 연결되어야 합니다.
6. 활동 날짜는 Activity가 원본입니다.
7. 캐릭터는 Activity가 원본입니다.
8. 사냥 시간은 Activity의 소요 시간(분)이 원본입니다.
9. 시작 레벨은 Activity의 활동 당시 레벨을 Rollup합니다.
10. 시작 레벨을 사냥 기록에 직접 중복 입력하지 않습니다.
11. 메소 측정 방식을 반드시 선택합니다.
12. 시작·종료 메소 방식에서는 시작 메소와 종료 메소를 모두 입력합니다.
13. 직접 획득 메소 방식에서는 직접 획득 메소를 입력합니다.
14. 선택하지 않은 측정 방식의 입력값은 비워두는 것을 원칙으로 합니다.
15. 소비 메소는 순수 메소 수익 계산에서 한 번만 차감합니다.
16. 소비 메소는 음수로 입력하지 않습니다.
17. 판매 아이템 예상 수익과 확정 수익을 구분합니다.
18. 아이템 판매 완료 시 확정 수익과 판매일을 입력합니다.
19. 예상 총수익과 확정 총수익은 서로 다른 목적에 사용합니다.
20. Activity의 사냥 예상 수익은 사냥 기록의 예상 총수익을 참조합니다.
21. Activity의 사냥 확정 수익은 사냥 기록의 확정 총수익을 참조합니다.
22. 시간당 메소는 확정 총수익을 기준으로 계산합니다.
23. 레벨업이 발생한 경험치 기록은 직접 경험치 증가율을 입력하거나 향후 자동 계산을 사용합니다.
24. 메소 입력 상태가 정상이 아닌 기록은 수익 집계 전에 검토합니다.
25. 판매 정산 상태가 정상이 아닌 기록은 확정 수익 집계 전에 검토합니다.
26. 연결 상태가 정상이 아닌 기록은 주간 및 Dashboard 집계 전에 검토합니다.
27. 사냥터 마스터의 현재 정보가 변경되어도 과거 사냥 기록의 수익은 변경되지 않습니다.
28. 완료된 사냥 기록은 삭제보다 수정과 보존을 우선합니다.

```text
2026-07-11 / 채집 재료 판매

2026-07-12 / 제작 아이템 판매
```

```text
예상 수입
- 지출
- 수수료
```

### 확정 순수익

```text
확정 수입
- 지출
- 수수료
```

### 시간당 확정 수익

```text
Activity의 소요 시간(분)이 0보다 크면

확정 순수익 ÷ Activity의 소요 시간(분) × 60

그렇지 않으면 0
```

### 총 거래 금액

필요 시 다음 Formula를 추가할 수 있습니다.

```text
수량 × 단가
```

## 24.9 경매장 거래 처리

초기 버전에서는 사용자가 실제 구매가와 판매가를 입력합니다.

예시:

```text
수입: 판매가
지출: 구매가
수수료: 경매장 수수료
확정 순수익: 판매가 - 구매가 - 수수료
```

메이플 옥션 공식 API가 제공되지 않는 경우 실시간 시세 자동 연동은 초기 범위에 포함하지 않습니다.

향후 시세 이력 DB나 외부 데이터 연동을 검토할 수 있습니다.

## 24.11 Rollup

| 속성명 | Relation | 원본 속성 | 설명 |

| 소요 시간(분) | Activity | 소요 시간(분) | 활동 시간 |
| 활동 유형 | Activity | 활동 유형 | 채집, 제작 등 |
| 활동 상태 | Activity | 활동 상태 | 완료 여부 확인 |

## 24.12 자동화 규칙

### 신규 기록 생성

1. Activity를 생성합니다.
2. 활동 유형을 설정합니다.
3. 기타 수익 기록을 생성합니다.
4. Activity와 상세 기록을 연결합니다.
5. 수입, 지출, 수수료를 입력합니다.
6. 예상 또는 확정 순수익을 계산합니다.

### 판매 완료 처리

1. 실제 판매 금액을 입력합니다.
2. 확정 수입을 갱신합니다.
3. 판매 완료를 체크합니다.
4. 거래 완료일을 기록합니다.
5. Activity의 기타 확정 수익 Rollup이 자동 갱신됩니다.

# 27. 거래 데이터 삭제 및 수정 정책

거래 상세 기록은 실제 플레이 이력을 나타내므로 삭제보다 수정을 우선합니다.

삭제를 검토할 수 있는 경우:

- 완전히 잘못 생성된 중복 기록
- 테스트용 기록
- 연결 실패로 생성된 빈 기록

과거 기록의 값이 잘못된 경우:

1. 기존 기록을 수정합니다.
2. 중요한 변경은 메모에 이유를 남깁니다.
3. API 자동화로 덮어쓰지 않도록 합니다.

# 28. Part 3 확정 사항

이번 Part에서 확정한 사항은 다음과 같습니다.

- Activity를 모든 플레이 활동의 공통 원장으로 사용합니다.
- Activity에는 날짜, 캐릭터, 활동 유형, 소요 시간 같은 공통 정보만 저장합니다.
- 보스, 사냥, 기타 수익 정보는 각각 별도의 상세 데이터베이스에 저장합니다.
- Activity 한 건은 원칙적으로 상세 기록 한 건과 연결합니다.
- Activity ID와 각 상세 기록 ID를 별도로 관리합니다.
- 수익은 예상 수익과 확정 수익으로 구분합니다.
- 실제 누적 수익에는 확정 수익만 반영합니다.
- 보스 결정석은 주간 관리에서 일괄 판매 여부를 관리합니다.
- 보스 드롭 수익은 예상 시세와 실제 판매 금액을 구분합니다.
- 사냥 수익은 시작·종료 메소 방식과 직접 획득 메소 방식을 지원합니다.
- 사냥 시간과 활동 날짜는 Activity가 원본입니다.
- 기타 수익 기록에서 채집, 제작, 경매장, 이벤트 및 지출을 관리합니다.
- 동일한 수익이나 지출을 여러 상세 DB에 중복 입력하지 않습니다.
- 완료된 거래 기록은 삭제보다 누적과 수정을 기본으로 합니다.
- Activity와 상세 기록 동시 생성을 향후 자동화 대상으로 확정합니다.

# Part 4. 요약 데이터베이스 및 Dashboard 설계

# 29. 요약 데이터 공통 원칙

요약 데이터베이스는 Activity와 거래 상세 데이터베이스를 기간 또는 목표 기준으로 집계합니다.

초기 요약 데이터베이스는 다음과 같습니다.

| DB 코드 | 데이터베이스명 | 역할 |
|---|---|---|
| WKL | 주간 관리 | 캐릭터별 주간 진행 현황과 수익 집계 |
| MSG | 목표 메소 | 목표 금액, 보유 메소 및 달성률 관리 |

Dashboard는 데이터베이스가 아니라 각 데이터베이스의 View와 계산 결과를 모아 보여주는 화면입니다.

---

## 29.1 요약 데이터 원칙

1. 거래 결과를 요약 데이터베이스에 다시 직접 입력하지 않습니다.
2. 주간 수익은 Activity와 거래 상세 기록에서 Rollup으로 집계합니다.
3. 완료율은 체크박스, Relation, Rollup, Formula로 계산합니다.
4. 동일한 기간과 캐릭터 조합의 주간 관리 기록은 하나만 존재해야 합니다.
5. 목표 메소는 실제 보유 메소와 예상 수익을 구분합니다.
6. Dashboard는 데이터 입력 공간이 아니라 조회 및 실행 공간으로 사용합니다.
7. 기간 기준 집계는 한국 메이플스토리의 주간 초기화 기준을 반영합니다.
8. 주간 관리 자동 생성과 초기화는 향후 Python 또는 GitHub Actions 자동화 대상으로 둡니다.

# 30. 주간 관리

## 30.1 기본 정보

| 항목 | 내용 |
|---|---|
| 데이터베이스명 | 주간 관리 |

---

## 30.2 목적

`주간 관리`는 캐릭터 한 명의 한 주 활동을 하나의 기록으로 관리합니다.

예시:

```text
2026-07-09 ~ 2026-07-15 / 본캐
2026-07-09 ~ 2026-07-15 / 보스돌이1
```

주간 관리에서는 다음 내용을 확인할 수 있습니다.

- 이번 주 보스 도전 및 클리어 상황
- 수로 완료 여부와 점수
- 플래그 완료 여부
- 결정석 일괄 판매 여부
- 몬스터파크 목표 및 완료 횟수
- 익스트림 몬스터파크 완료 여부
- 에픽 던전 완료 여부
- 주간 활동 수
- 주간 플레이 시간
- 주간 예상 수익
- 주간 확정 수익
- 주간 완료율
- 미완료 항목
- 마감 당시 주간 실적

---

## 30.3 기록 단위

캐릭터 한 명당 한 주에 하나의 기록을 생성합니다.

고유 조합:

```text
캐릭터 + 주차 시작일
```

동일 캐릭터와 동일 주차 시작일을 가진 기록은 중복 생성하지 않습니다.

```text
WKL-20260709-CHR0001
WKL-20260709-CHR0002
```

```text
WKL-주차시작일-캐릭터ID
```

### 기록명

```text
주차 기간 / 캐릭터명
```

예시:

```text
2026-07-09 ~ 2026-07-15 / 본캐
```

## 30.6 주차 운영 상태 옵션

```text
예정
진행 중
마감
```

### 예정

아직 주차가 시작되지 않은 기록입니다.

### 진행 중

현재 활동을 기록하고 콘텐츠를 수행하는 주차입니다.

### 마감

주차가 종료되어 해당 주의 실적을 확정한 상태입니다.

`주차 운영 상태`는 기록의 시간적·관리적 상태를 나타냅니다.

콘텐츠 완료 여부는 별도의 `주간 완료 상태` Formula에서 판단합니다.

---

## 30.7 캐릭터 설정 Rollup

주간 완료 여부는 캐릭터별 관리 설정을 기준으로 판단합니다.

따라서 캐릭터 Relation을 통해 다음 값을 Rollup합니다.

| 속성명 | 원본 DB | 원본 속성 | 설명 |
|---|---|---|---|
| 수로 관리 대상 | 캐릭터 | 수로 관리 | 수로 완료 여부가 필요한지 판단 |
| 플래그 관리 대상 | 캐릭터 | 플래그 관리 | 플래그 완료 여부가 필요한지 판단 |
| 결정석 관리 대상 | 캐릭터 | 결정석 관리 | 결정석 판매 완료 여부가 필요한지 판단 |
| 몬스터파크 관리 대상 | 캐릭터 | 몬스터파크 관리 | 몬스터파크 완료 여부가 필요한지 판단 |
| 익몬 관리 대상 | 캐릭터 | 익스트림 몬스터파크 관리 | 익몬 완료 여부 판단 |
| 에픽 던전 관리 대상 | 캐릭터 | 에픽 던전 관리 | 에픽 던전 완료 여부 판단 |
| 보스 관리 대상 | 캐릭터 | 보스 관리 | 보스 완료율 계산 대상 여부 |
| 현재 관리 대상 | 캐릭터 | 현재 관리 대상 | 주간 관리 유지 여부 판단 |

현재 버전에서는 캐릭터의 현재 관리 설정을 Rollup하여 사용합니다.

따라서 캐릭터 마스터의 관리 설정이 변경되면 과거 주간 기록의 필수 항목 수와 완료율도 변경될 수 있습니다.

향후 Python 자동화가 구축되면 주간 기록 생성 당시의 관리 설정을 별도 체크박스 속성에 복사하는 `관리 설정 스냅샷` 구조를 추가합니다.

스냅샷 도입 이후에는 주간 완료율 계산 기준을 현재 설정 Rollup에서 주차별 스냅샷으로 변경합니다.

## 30.8 Activity Rollup

Activity에 활동 유형별 보조 Formula를 두고 주간 관리에서 이를 Rollup합니다.

| 속성명 | 원본 속성 | 계산 방식 | 설명 |
|---|---|---|---|
| 주간 활동 수 | Activity의 활동명 | 개수 | 해당 주 전체 활동 수 |
| 주간 플레이 시간(분) | Activity의 소요 시간(분) | 합계 | 해당 주 전체 플레이 시간 |
| 주간 예상 수익 | Activity의 예상 수익 | 합계 | 판매 전 예상 수익을 포함한 전체 예상 수익 |
| 주간 확정 수익 | Activity의 확정 수익 | 합계 | 실제 확정된 전체 수익 |
| 주간 보스 도전 수 | Activity의 보스 도전 수 | 합계 | 실패, 중단, 연습을 포함한 전체 보스 도전 횟수 |
| 주간 보스 클리어 수 | Activity의 보스 클리어 수 | 합계 | 실제 클리어한 보스 활동 수 |
| 주간 보스 확정 수익 | Activity의 보스 확정 수익 | 합계 | 보스 활동에서 발생한 확정 수익 |
| 주간 결정석 총액 | Activity의 보스 결정석 수익 | 합계 | 보스 클리어로 확정된 개인 결정석 수익 |
| 주간 사냥 활동 수 | Activity의 사냥 활동 수 | 합계 | 사냥 세션 수 |
| 주간 사냥 시간(분) | Activity의 사냥 시간(분) | 합계 | 사냥 활동에 사용한 시간 |
| 주간 사냥 확정 수익 | Activity의 사냥 확정 수익 | 합계 | 사냥 활동에서 발생한 확정 수익 |
| 주간 기타 확정 수익 | Activity의 기타 확정 수익 | 합계 | 기타 활동에서 발생한 확정 수익 |
| 최근 활동 날짜 | Activity의 활동 날짜 | 가장 최근 날짜 | 마지막으로 기록된 활동일 |

보스 도전 수와 보스 클리어 수는 서로 구분합니다.

```text
보스 도전 수
→ 활동 유형이 보스이면 1

보스 클리어 수
→ 활동 유형이 보스이고 연결된 보스 기록의 클리어 여부가 참이면 1
```

실패, 중단, 연습 기록은 보스 도전 수에는 포함하지만 보스 클리어 수에는 포함하지 않습니다.

## 30.9 보스 완료율 구조

실제 수행한 보스 활동 수만으로는 전체 목표 보스 수를 자동으로 알 수 없습니다.

따라서 향후 `캐릭터별 보스 설정` 데이터베이스를 추가하는 것이 가장 정확합니다.

예상 구조:

```text
캐릭터
    │
    └── 캐릭터별 보스 설정
            │
            ├── 보스
            ├── 난이도
            ├── 주간 목표 여부
            └── 활성 여부
```

### v0.2 초기 구현

주간 관리의 `목표 보스 수` 속성에 해당 주의 목표 보스 수를 직접 입력합니다.

완료 보스 수는 Activity에서 Rollup한 `주간 보스 클리어 수`를 사용합니다.

```text
목표 보스 수
→ 사용자가 직접 입력하거나 자동화가 입력

완료 보스 수
→ 주간 보스 클리어 수 Rollup
```

동일한 보스를 여러 번 클리어하거나 중복 기록하면 단순 클리어 건수가 목표 보스 수보다 많아질 수 있습니다.

따라서 보스 완료율은 최대 100%로 제한합니다.

### 장기 구현

`캐릭터별 보스 설정` 데이터베이스의 활성화된 주간 목표 보스 수를 Rollup합니다.

장기적으로는 단순 클리어 건수가 아니라 목표로 설정된 개별 보스의 완료 여부를 기준으로 계산합니다.

---

## 30.10 계산 및 집계 속성

| 속성명 | Notion 유형 | 설명 |
|---|---|---|
| 완료 보스 수 | Rollup | Activity에서 Rollup한 주간 보스 클리어 수 |
| 보스 완료율 | Formula | 완료 보스 수 ÷ 목표 보스 수, 최대 100% |
| 보스 목표 설정 상태 | Formula | 보스 목표 설정 완료 또는 보스 목표 미설정 표시 |
| 몬스터파크 완료 여부 | Formula | 완료 횟수가 목표 횟수 이상인지 판단 |
| 결정석 항목 완료 여부 | Formula | 결정석 총액과 판매 완료 여부를 기준으로 판단 |
| 필수 항목 수 | Formula | 활성화되고 정상 설정된 주간 관리 항목 수 |
| 완료 항목 수 | Formula | 실제 완료된 필수 항목 수 |
| 주간 완료율 | Formula | 완료 항목 수 ÷ 필수 항목 수 |
| 주간 완료 상태 | Formula | 관리 항목 없음, 목표 미설정, 진행 중, 완료 표시 |
| 미확정 예상 수익 | Formula | 예상 수익 중 아직 확정되지 않은 금액 |
| 시간당 확정 수익 | Formula | 주간 확정 수익 ÷ 주간 플레이 시간 |
| 미완료 항목 | Formula | 아직 완료하지 않은 필수 콘텐츠 목록 |

## 30.11 Formula

### 주차 종료일

```text
주차 시작일이 입력되어 있으면

주차 시작일 + 6일

그렇지 않으면 빈 값
```

### 보스 완료율

```text
목표 보스 수가 0보다 크면

완료 보스 수 ÷ 목표 보스 수와 1 중 작은 값

그렇지 않으면 0
```

보스 완료율은 최대 100%로 제한합니다.

### 보스 목표 설정 상태

```text
보스 관리 대상이 아니면 "관리 대상 아님"

보스 관리 대상이고 목표 보스 수가 0보다 크면 "설정 완료"

그렇지 않으면 "보스 목표 미설정"
```

### 몬스터파크 완료 여부

```text
몬스터파크 관리 대상이 아니면 거짓

몬스터파크 목표 횟수가 0보다 크고
몬스터파크 완료 횟수가 목표 횟수 이상이면 참

그렇지 않으면 거짓
```

몬스터파크 목표 횟수가 설정되지 않은 경우 완료로 처리하지 않습니다.

### 결정석 항목 완료 여부

```text
결정석 관리 대상이 아니면 거짓

주간 결정석 총액이 0이면 참

주간 결정석 총액이 0보다 크고
결정석 일괄 판매 완료가 체크되어 있으면 참

그렇지 않으면 거짓
```

판매할 결정석이 없는 주차는 결정석 항목을 자동 완료로 판단합니다.

### 필수 항목 수

개념식:

```text
수로 관리 대상이면 1

+ 플래그 관리 대상이면 1

+ 결정석 관리 대상이면 1

+ 몬스터파크 관리 대상이고
  몬스터파크 목표 횟수가 0보다 크면 1

+ 익몬 관리 대상이면 1

+ 에픽 던전 관리 대상이면 1

+ 보스 관리 대상이고
  목표 보스 수가 0보다 크면 1
```

목표 값이 필요한 콘텐츠가 설정되지 않은 경우 해당 항목은 필수 항목 수에 포함하지 않습니다.

설정 누락은 `주간 완료 상태`에서 별도로 표시합니다.

### 완료 항목 수

개념식:

```text
수로 관리 대상이고 수로 완료이면 1

+ 플래그 관리 대상이고 플래그 완료이면 1

+ 결정석 관리 대상이고 결정석 항목 완료 여부가 참이면 1

+ 몬스터파크 관리 대상이고
  몬스터파크 완료 여부가 참이면 1

+ 익몬 관리 대상이고
  익스트림 몬스터파크 완료이면 1

+ 에픽 던전 관리 대상이고
  에픽 던전 완료이면 1

+ 보스 관리 대상이고
  목표 보스 수가 0보다 크고
  보스 완료율이 100%이면 1
```

### 주간 완료율

```text
필수 항목 수가 0보다 크면

완료 항목 수 ÷ 필수 항목 수

그렇지 않으면 0
```

### 주간 완료 상태

```text
보스 관리 대상이고 목표 보스 수가 0이면
"보스 목표 미설정"

몬스터파크 관리 대상이고
몬스터파크 목표 횟수가 0이면
"몬스터파크 목표 미설정"

필수 항목 수가 0이면
"관리 항목 없음"

주간 완료율이 100%이면
"완료"

그렇지 않으면
"진행 중"
```

여러 목표가 동시에 설정되지 않은 경우 Formula 구현 단계에서 `목표 미설정`으로 통합하거나 누락 항목을 함께 표시할 수 있습니다.

### 미확정 예상 수익

```text
주간 예상 수익 - 주간 확정 수익이 0보다 크면

주간 예상 수익 - 주간 확정 수익

그렇지 않으면 0
```

실제 확정 수익이 예상 수익보다 큰 경우 음수로 표시하지 않습니다.

### 시간당 확정 수익

```text
주간 플레이 시간(분)이 0보다 크면

주간 확정 수익 ÷ 주간 플레이 시간(분) × 60

그렇지 않으면 0
```

### 미완료 항목

개념식:

```text
수로 관리 대상이고 수로 미완료이면 "수로"

플래그 관리 대상이고 플래그 미완료이면 "플래그"

결정석 관리 대상이고
결정석 항목 완료 여부가 거짓이면 "결정석 판매"

몬스터파크 관리 대상이고
몬스터파크 완료 여부가 거짓이면 "몬스터파크"

익몬 관리 대상이고
익스트림 몬스터파크 미완료이면 "익몬"

에픽 던전 관리 대상이고
에픽 던전 미완료이면 "에픽 던전"

보스 관리 대상이고
목표 보스 수가 0이면 "보스 목표 설정"

보스 관리 대상이고
목표 보스 수가 0보다 크고
보스 완료율이 100% 미만이면 "보스"
```

미완료 항목이 여러 개인 경우 쉼표로 구분하여 표시합니다.

## 30.12 결정석 일괄 판매 구조

결정석 수익 발생과 NPC 판매 완료 여부를 분리하여 관리합니다.

처리 흐름:

```text
보스 클리어
    │
    ▼
보스 기록에 개인 결정석 수익 확정
    │
    ▼
Activity 확정 수익에 즉시 반영
    │
    ▼
Activity의 보스 결정석 수익에 반영
    │
    ▼
주간 관리에서 주간 결정석 총액 Rollup
    │
    ▼
NPC 일괄 판매
    │
    ▼
결정석 일괄 판매 완료 체크
    │
    ▼
결정석 판매일 입력
```

결정석 금액은 보스 클리어 시점에 확정 수익으로 집계합니다.

`결정석 일괄 판매 완료` 체크는 메소 수익 발생 여부를 변경하지 않습니다.

이 체크박스는 결정석을 실제 NPC에게 판매했는지를 관리하기 위한 운영 상태입니다.

주간 결정석 총액이 0이면 판매할 결정석이 없는 것으로 판단하여 결정석 항목을 자동 완료 처리합니다.

주간 결정석 총액이 0보다 크면 `결정석 일괄 판매 완료`가 체크되어야 완료로 판단합니다.

## 30.13 마감 스냅샷

Formula와 Rollup 값은 원본 데이터가 변경되면 다시 계산됩니다.

따라서 주차 마감 당시의 결과를 보존하기 위해 다음 스냅샷 속성을 사용합니다.

| 속성명 | Notion 유형 | 입력 주체 | 설명 |
|---|---|---|---|

| 마감 당시 미완료 항목 | 텍스트 | 자동화 | 마감 시점에 완료하지 못한 항목 |

초기 수동 운영 단계에서는 마감 스냅샷 속성을 비워둘 수 있습니다.

Python 자동화가 구축된 이후 주차 마감 시 Formula와 Rollup 결과를 일반 속성에 복사하여 고정합니다.

## 30.14 자동화 규칙

### 주간 관리 자동 생성

매주 주간 초기화 시점에 다음 조건의 캐릭터를 대상으로 생성합니다.

```text
현재 관리 대상 = 체크
주간 관리 사용 = 체크
캐릭터 상태가 활성 또는 육성 중
자동 주간 생성 = 체크
```

생성 과정:

1. 캐릭터별 주간 관리 기록 존재 여부 확인
2. 동일한 캐릭터와 주차 시작일 조합이 없으면 새 기록 생성
3. 주간 관리 ID 생성
4. 주차 시작일 입력
5. 주차 종료일 Formula 자동 계산
6. 캐릭터 연결
7. 주차 운영 상태를 진행 중으로 설정
8. 캐릭터 관리 설정을 Rollup으로 가져오기
9. 몬스터파크 목표 횟수 입력
10. 목표 보스 수 입력 또는 보스 설정 연결
11. 해당 캐릭터와 주차의 Activity 자동 연결 준비

### 주차 마감

주차 종료 후 다음 순서로 마감합니다.

1. 해당 주차의 Activity 연결 상태 확인
2. 주간 완료율과 미완료 항목 계산
3. 마감 당시 주간 완료율 저장
4. 마감 당시 예상 수익 저장
5. 마감 당시 확정 수익 저장
6. 마감 당시 미확정 예상 수익 저장
7. 마감 당시 미완료 항목 저장
8. 마감 당시 보스 완료율 저장
9. 마감 당시 주간 플레이 시간 저장
10. 마감일시 입력
11. 주차 운영 상태를 마감으로 변경
12. 다음 주차 기록 생성

마감 스냅샷은 Formula와 Rollup 값을 일반 숫자, 텍스트, 날짜 속성에 복사하는 방식으로 저장합니다.

마감 이후 Activity 수정 제한 기능은 향후 자동화에서 검토합니다.

## 30.16 데이터 무결성 규칙

1. 동일 캐릭터와 동일 주차 시작일의 기록은 중복될 수 없습니다.
2. 모든 주간 관리 기록은 하나의 캐릭터와 연결되어야 합니다.
3. 주차 종료일은 주차 시작일을 기준으로 자동 계산합니다.
4. 주간 관리에는 해당 캐릭터의 해당 주 Activity만 연결해야 합니다.
5. 관리 대상이 아닌 항목은 필수 항목과 미완료 항목에 포함하지 않습니다.
6. 목표 값이 필요한 항목은 목표가 설정된 경우에만 필수 항목 수에 포함합니다.
7. 보스 완료 수는 보스 도전 수가 아니라 실제 클리어 수를 기준으로 계산합니다.
8. 실패, 중단, 연습 보스 기록은 완료 보스 수에 포함하지 않습니다.
9. 보스 완료율은 100%를 초과할 수 없습니다.
10. 동일 보스의 중복 클리어 기록은 초기 버전에서 단순 클리어 건수로 집계될 수 있습니다.
11. 장기적으로는 캐릭터별 보스 설정을 통해 개별 목표 보스 완료 여부를 관리합니다.
12. 결정석 수익은 보스 클리어 시점에 확정 수익으로 집계합니다.
13. 결정석 일괄 판매 여부는 수익 발생 여부를 변경하지 않습니다.
14. 주간 결정석 총액이 0이면 결정석 판매 항목은 자동 완료로 판단합니다.
15. 몬스터파크 완료 횟수는 0 미만일 수 없습니다.
16. 몬스터파크 완료 여부는 목표 횟수 이상을 달성했을 때만 참으로 계산합니다.
17. 미확정 예상 수익은 0 미만으로 계산하지 않습니다.
18. `주차 운영 상태`와 `주간 완료 상태`는 서로 다른 의미로 사용합니다.
19. 주차 운영 상태는 예정, 진행 중, 마감만 사용합니다.
20. 마감된 기록도 삭제하지 않습니다.
21. 마감 스냅샷 값은 마감 이후 원본 Formula 및 Rollup 변경과 무관하게 유지합니다.
22. 현재 버전의 관리 대상 여부는 캐릭터의 현재 설정을 Rollup하여 계산합니다.
23. 캐릭터 관리 설정이 변경되면 과거 주간 기록의 계산값도 변경될 수 있습니다.
24. 향후 관리 설정 스냅샷 도입 시 과거 주간 기록은 주차 생성 당시 설정을 기준으로 계산합니다.
25. 자동화는 동일 캐릭터와 동일 주차 시작일의 기록이 존재하면 새 기록을 생성하지 않습니다.

## 31.8 Formula

### 부족한 메소

```text
목표 금액 - 현재 보유 메소
```

결과가 0보다 작으면 0으로 표시합니다.

### 예상 부족 메소

```text
목표 금액 - 현재 보유 메소 - 확보 예정 메소
```

결과가 0보다 작으면 0으로 표시합니다.

### 실제 진행률

```text
목표 금액이 0보다 크면

현재 보유 메소 ÷ 목표 금액

그렇지 않으면 0
```

진행률이 100%를 초과하면 100%로 제한할 수 있습니다.

### 예상 진행률

```text
목표 금액이 0보다 크면

(현재 보유 메소 + 확보 예정 메소) ÷ 목표 금액

그렇지 않으면 0
```

### 남은 일수

```text
목표 날짜가 존재하면

목표 날짜 - 오늘

그렇지 않으면 빈 값
```

### 목표 상태 자동 안내

개념식:

```text
완료 여부가 체크되어 있으면 "완료"

목표 날짜가 지났고 완료되지 않았으면 "기한 초과"

실제 진행률이 100%이면 "완료 가능"

그 외에는 목표 상태 표시
```

## 31.9 현재 보유 메소 관리 원칙

현재 보유 메소는 수익 누적과 동일하지 않습니다.

예를 들어 다음 상황이 발생할 수 있습니다.

- 메소를 획득했지만 아이템 구매에 사용함
- 다른 캐릭터나 월드로 이동함
- 경매장에 등록되어 아직 정산되지 않음
- 판매 예정 아이템이 있음

따라서 목표 메소의 `현재 보유 메소`는 실제 보유 금액을 기준으로 관리합니다.

Activity의 누적 수익을 그대로 현재 보유 메소로 사용하지 않습니다.

향후 `메소 잔액 기록` 또는 `자산 스냅샷` 데이터베이스를 추가하면 자동 갱신할 수 있습니다.

# 35. Activity Formula 보조 속성

캐릭터, 주간 관리 및 Dashboard에서 활동 유형별 통계를 Rollup하기 위해 Activity에 내부 계산용 보조 속성을 둡니다.

이 보조 속성들은 사용자가 직접 입력하지 않습니다.

Activity에서 유형별 값을 먼저 계산하고, 캐릭터와 주간 관리에서는 해당 값을 합계 Rollup하는 구조를 사용합니다.

```text
Activity에서 유형별 Formula 계산
→ 캐릭터 및 주간 관리에서 합계 Rollup
→ Dashboard에서 통계 표시
```

---

## 35.1 활동 수 보조 Formula

| 속성명 | Notion 유형 | 개념식 | 주요 사용처 |
|---|---|---|---|
| 보스 도전 수 | Formula | 실제 시작된 보스 활동이면 1, 아니면 0 | 주간 및 누적 보스 도전 횟수 |
| 보스 클리어 수 | Formula | 보스 활동이고 실제 클리어했으면 1, 아니면 0 | 완료 보스 수 및 보스 완료율 |
| 사냥 활동 수 | Formula | 사냥 활동이고 완료 상태이면 1, 아니면 0 | 주간 및 누적 사냥 세션 수 |
| 기타 활동 수 | Formula | 보스와 사냥이 아닌 활동이고 완료 상태이면 1, 아니면 0 | 기타 콘텐츠 활동 수 |
| 완료 활동 수 | Formula | 완료 여부가 참이면 1, 아니면 0 | 전체 완료 활동 수 |

---

## 35.2 보스 도전 수

보스 Activity가 생성되었다는 이유만으로 도전 수에 포함하지 않습니다.

예정 상태이거나 취소된 기록은 실제 도전으로 간주하지 않습니다.

개념식:

```text
활동 유형이 "보스"이고

활동 상태가 "진행 중" 또는 "완료"이면

1

그렇지 않으면 0
```

다음 기록은 보스 도전 수에 포함합니다.

- 클리어
- 실패
- 중단
- 연습
- 진행 중인 실제 도전

다음 기록은 보스 도전 수에 포함하지 않습니다.

- 예정
- 취소
- 잘못 생성된 빈 기록

보스 기록의 `도전 결과`가 `미진행`인 경우에는 Activity 상태가 완료더라도 도전 수에서 제외하는 것이 가장 정확합니다.

이를 위해 Activity에 보스 기록의 `도전 결과`를 Rollup하는 속성을 추가할 수 있습니다.

### 보스 도전 결과 Rollup

| 속성명 | Notion 유형 | 원본 속성 | 설명 |
|---|---|---|---|
| 보스 도전 결과 | Rollup | 보스 기록의 도전 결과 | 실제 보스 도전 결과 확인용 |

`보스 도전 결과` Rollup을 사용할 경우 권장 개념식은 다음과 같습니다.

```text
활동 유형이 "보스"이고

활동 상태가 "예정" 또는 "취소"가 아니고

보스 도전 결과가 "미진행"이 아니면

1

그렇지 않으면 0
```

초기 Notion 구현에서 Rollup 비교가 불편하면 활동 상태만 기준으로 계산하고, Python 자동화에서 `미진행` 기록을 검증합니다.

---

## 35.3 보스 클리어 수

`보스 클리어 수`는 Activity의 완료 여부가 아니라 보스 기록의 실제 `클리어 여부`를 기준으로 계산합니다.

개념식:

```text
활동 유형이 "보스"이고

보스 클리어 여부가 참이면

1

그렇지 않으면 0
```

다음 기록은 보스 클리어 수에 포함하지 않습니다.

- 실패
- 중단
- 연습
- 미진행
- 예정
- 취소
- 보스 기록 미연결
- 보스 클리어 여부가 거짓인 기록

보스 Activity 자체가 완료되어 있더라도 실제 클리어하지 못했다면 보스 클리어 수는 0입니다.

예시:

```text
활동 상태: 완료
완료 여부: TRUE
도전 결과: 실패
보스 클리어 여부: FALSE
보스 도전 수: 1
보스 클리어 수: 0
```

---

## 35.4 사냥 활동 수

개념식:

```text
활동 유형이 "사냥"이고

완료 여부가 참이면

1

그렇지 않으면 0
```

진행 중인 사냥 기록은 아직 완료된 세션으로 집계하지 않습니다.

사냥 시작 횟수와 사냥 완료 횟수를 별도로 분석할 필요가 생기면 향후 다음 속성을 추가할 수 있습니다.

```text
사냥 시작 수
사냥 완료 수
```

초기 버전에서는 `사냥 활동 수`를 완료된 사냥 세션 수로 사용합니다.

---

## 35.5 기타 활동 수

개념식:

```text
활동 유형이 "보스" 또는 "사냥"이 아니고

완료 여부가 참이면

1

그렇지 않으면 0
```

기타 활동에는 초기 버전 기준으로 다음 유형이 포함될 수 있습니다.

- 채집
- 제작
- 경매장
- 이벤트
- 몬스터파크
- 익스트림 몬스터파크
- 에픽 던전
- 무릉
- 수로
- 플래그
- 기타

향후 전용 상세 기록 DB가 추가되어도 해당 활동이 보스나 사냥이 아니라면 기본적으로 기타 활동 수에 포함할 수 있습니다.

---

## 35.6 완료 활동 수

개념식:

```text
완료 여부가 참이면 1

그렇지 않으면 0
```

`완료 활동 수`는 Activity의 처리 상태를 기준으로 계산합니다.

보스 활동에서는 실제 보스 클리어 여부와 다를 수 있습니다.

```text
보스 실패 기록
→ Activity 완료 여부: TRUE
→ 완료 활동 수: 1
→ 보스 클리어 수: 0
```

---

## 35.7 수익 보조 속성

| 속성명 | Notion 유형 | 개념식 또는 원본 | 주요 사용처 |
|---|---|---|---|
| 보스 확정 수익 | Formula | 활동 유형이 보스이면 확정 수익, 아니면 0 | 주간 및 누적 보스 수익 |
| 사냥 확정 수익 | Formula | 활동 유형이 사냥이면 확정 수익, 아니면 0 | 주간 및 누적 사냥 수익 |
| 기타 확정 수익 | Formula | 활동 유형이 보스와 사냥이 아니면 확정 수익, 아니면 0 | 주간 및 누적 기타 수익 |
| 보스 결정석 수익 | Rollup | 보스 기록의 개인 결정석 수익 | 주간 결정석 총액 및 누적 결정석 수익 |

---

## 35.8 보스 확정 수익

개념식:

```text
활동 유형이 "보스"이면

확정 수익

그렇지 않으면 0
```

`보스 확정 수익`에는 다음 값이 포함될 수 있습니다.

- 개인 결정석 수익
- 개인 확정 드롭 수익
- 보스 소비 비용 차감 결과

정확한 값은 보스 기록의 `확정 총수익`에서 가져온 Activity의 `확정 수익`을 기준으로 합니다.

---

## 35.9 사냥 확정 수익

개념식:

```text
활동 유형이 "사냥"이면

확정 수익

그렇지 않으면 0
```

사냥 기록의 `확정 총수익`을 Activity의 `사냥 확정 수익` Rollup으로 가져옵니다.

Activity에서는 사냥 기록에서 최종 계산된 확정 수익만 참조하며, 순수 메소 수익이나 판매 아이템 수익을 다시 계산하지 않습니다.

---

## 35.10 기타 확정 수익

개념식:

```text
활동 유형이 "보스" 또는 "사냥"이 아니면

확정 수익

그렇지 않으면 0
```

기타 수익에는 양수 수익뿐 아니라 지출로 인한 음수 수익도 포함될 수 있습니다.

예시:

```text
아이템 판매
→ 양수

아이템 구매
→ 음수

강화 비용
→ 음수
```

따라서 기타 확정 수익은 음수 값을 허용합니다.

---

## 35.11 보스 결정석 수익

`보스 결정석 수익`은 Activity에서 새로 계산하는 Formula가 아닙니다.

보스 기록의 `개인 결정석 수익`을 가져오는 Rollup 속성입니다.

```text
보스 기록의 개인 결정석 수익

→ Activity의 보스 결정석 수익 Rollup

→ 주간 관리의 주간 결정석 총액 Rollup

→ 캐릭터의 누적 결정석 수익 Rollup
```

보스 결정석 수익에는 다음 금액을 포함하지 않습니다.

- 개인 예상 드롭 수익
- 개인 확정 드롭 수익
- 파티 드롭 분배 수익
- 솔 에르다의 기운 등 개인 보상
- 기타 보상 수익

결정석 수익과 전체 보스 수익은 서로 다른 목적으로 사용합니다.

```text
보스 확정 수익
→ 보스에서 발생한 전체 확정 수익

보스 결정석 수익
→ 결정석 수익만 별도로 집계
```

---

## 35.12 시간 보조 Formula

| 속성명 | Notion 유형 | 개념식 | 주요 사용처 |
|---|---|---|---|
| 보스 시간(분) | Formula | 활동 유형이 보스이면 소요 시간(분), 아니면 0 | 보스 플레이 시간 분석 |
| 사냥 시간(분) | Formula | 활동 유형이 사냥이면 소요 시간(분), 아니면 0 | 주간 및 누적 사냥 시간 |
| 기타 시간(분) | Formula | 보스와 사냥이 아니면 소요 시간(분), 아니면 0 | 기타 콘텐츠 시간 분석 |

### 보스 시간(분)

```text
활동 유형이 "보스"이면

소요 시간(분)

그렇지 않으면 0
```

### 사냥 시간(분)

```text
활동 유형이 "사냥"이면

소요 시간(분)

그렇지 않으면 0
```

### 기타 시간(분)

```text
활동 유형이 "보스" 또는 "사냥"이 아니면

소요 시간(분)

그렇지 않으면 0
```

소요 시간(분)이 비어 있는 경우에는 0으로 처리합니다.


## 35.16 집계 제외 기준

다음 Activity는 통계 집계 전에 검토하거나 제외합니다.

- 연결 오류 상태가 정상이지 않은 Activity
- 캐릭터가 연결되지 않은 Activity
- 활동 유형이 비어 있는 Activity
- 상세 기록이 중복 연결된 Activity
- 활동 유형과 상세 기록 유형이 불일치한 Activity
- 테스트용 Activity
- 잘못 생성된 빈 Activity
- 취소된 Activity

취소 상태의 Activity는 기본적으로 활동 수, 시간 및 수익 집계에서 제외합니다.

취소된 Activity에 상세 기록이나 수익이 남아 있다면 Python 자동화에서 오류로 표시합니다.

---

## 35.17 Formula 구현 시 공통 처리

각 보조 Formula는 다음 원칙을 따릅니다.

1. 비어 있는 숫자는 0으로 처리합니다.
2. 취소 상태의 Activity는 집계하지 않습니다.
3. 예정 상태는 완료 통계에 포함하지 않습니다.
4. 보스 클리어 수는 보스 기록의 클리어 여부를 기준으로 합니다.
5. Activity 완료 여부와 보스 클리어 여부를 혼합하지 않습니다.
6. 수익 유형별 Formula는 Activity의 최종 예상 수익과 확정 수익을 기준으로 합니다.
7. 보스 결정석 수익은 Formula가 아니라 Rollup으로 가져옵니다.
8. 시간 보조 속성은 Activity의 소요 시간(분)을 기준으로 합니다.
9. 연결 오류가 있는 Activity는 Python 자동화에서 별도로 검증합니다.
10. Dashboard에서는 상세 DB를 직접 다시 계산하지 않고 Activity 보조 속성을 집계합니다.

---

## 35.18 데이터 무결성 규칙

1. 보스 도전 수와 보스 클리어 수를 서로 같은 값으로 사용하지 않습니다.
2. 실패, 중단, 연습은 보스 도전 수에는 포함할 수 있지만 보스 클리어 수에는 포함하지 않습니다.
3. 예정 및 취소 상태의 보스 Activity는 보스 도전 수에서 제외합니다.
4. `도전 결과 = 미진행`인 보스 기록은 보스 도전 수에서 제외하는 것을 원칙으로 합니다.
5. 사냥 활동 수는 완료된 사냥 세션만 집계합니다.
6. 기타 활동 수는 보스와 사냥이 아닌 완료 Activity를 기준으로 계산합니다.
7. 완료 활동 수는 Activity의 완료 여부를 기준으로 계산합니다.
8. 보스 확정 수익에는 결정석과 확정 드롭 수익이 포함될 수 있습니다.
9. 보스 결정석 수익에는 드롭 수익을 포함하지 않습니다.
10. 기타 확정 수익은 지출로 인해 음수가 될 수 있습니다.
11. 시간 보조 속성은 소요 시간(분)을 중복 저장하지 않고 Formula로 분류합니다.
12. 캐릭터와 주간 관리에서는 Activity 보조 속성을 합계 Rollup합니다.
13. 같은 의미의 속성을 캐릭터, 주간 관리 또는 Dashboard에서 다시 Formula로 계산하지 않습니다.
14. 연결 오류 상태가 정상이지 않은 Activity는 통계 신뢰성 검토 대상입니다.
15. 보조 속성은 사용자가 직접 수정하지 않습니다.

---

# 36. Formula 구현 원칙

1. 0으로 나누는 오류를 반드시 방지합니다.
2. 비어 있는 숫자 속성은 0으로 처리합니다.
3. 진행률은 가능하면 0%에서 100% 사이로 제한합니다.
4. 예상 수익과 확정 수익을 혼합하지 않습니다.
5. 완료 여부와 상태값이 서로 모순되지 않도록 합니다.
6. Formula 속성명은 결과를 명확하게 표현합니다.
7. 복잡한 Formula는 개념식과 실제 Notion Formula를 문서에 함께 기록합니다.
8. Notion Formula 한계가 있는 검증은 Python 자동화에서 처리합니다.

---

# 37. Notion View 공통 원칙

1. 원본 데이터베이스마다 `전체` View를 유지합니다.
2. Dashboard용 View는 별도로 만듭니다.
3. 기간 View는 오늘, 이번 주, 이번 달 기준으로 구성합니다.
4. 상태 View는 미완료, 완료, 오류 중심으로 구성합니다.
5. 불필요한 속성은 View에서 숨깁니다.
6. 수익 View는 확정 수익을 기본 정렬 기준으로 사용합니다.
7. 예상 수익은 미판매 자산 확인용 View에서 사용합니다.
8. 마스터 DB는 사용 여부가 체크된 항목을 기본 표시합니다.
9. 거래 기록은 날짜 내림차순으로 정렬합니다.
10. 요약 DB는 현재 기간을 기본 표시합니다.

# 39. 초기 구현 검증 기준

각 데이터베이스 생성 후 다음 사항을 확인합니다.

## 캐릭터

- 캐릭터 ID가 존재하는가
- API 속성과 사용자 설정 속성이 분리되어 있는가
- 캐릭터 상태와 관리 기능이 정상적으로 설정되는가

## Activity

- 캐릭터와 연결되어 있는가
- 활동 유형이 설정되어 있는가
- 상세 기록 하나만 연결되어 있는가
- 예상 수익과 확정 수익이 정상 집계되는가

## 보스 기록

- 보스 마스터와 연결되어 있는가
- 결정석 가격 스냅샷이 저장되는가
- 예상 드롭과 확정 드롭이 구분되는가

## 사냥 기록

- Activity와 사냥터가 연결되는가
- 시간당 메소 계산이 정상인가
- 시작·종료 메소와 직접 획득 메소 우선순위가 정상인가

## 기타 수익 기록

- 수입, 지출, 수수료가 중복 계산되지 않는가
- 예상 수입과 확정 수입이 구분되는가

## 주간 관리

- 캐릭터별 관리 설정을 반영하는가
- 관리하지 않는 항목이 미완료로 표시되지 않는가
- 해당 주 Activity만 집계되는가
- 결정석 일괄 판매 체크가 정상인가

## 목표 메소

- 실제 보유 메소와 예상 확보 메소가 분리되어 있는가
- 부족한 메소와 진행률이 정상인가

## Dashboard

- 원본 데이터를 중복 저장하지 않는가
- 미완료와 최근 활동이 쉽게 보이는가
- 입력 버튼과 조회 영역이 구분되는가

2. 속성 이름 변경 시 API 코드와 Notion 구조 영향을 함께 검토합니다.

4. 중요한 변경은 Git Commit Extended description에 목적을 기록합니다.

6. 과거 설계 결정을 변경할 때는 이유를 남깁니다.
7. 초기 구현 완료 후 문서 상태를 Draft에서 Approved로 변경합니다.
8. 문서 버전은 구조 변경 규모에 따라 갱신합니다.

# 44. Part 4 확정 사항

이번 Part에서 확정한 사항은 다음과 같습니다.

- 주간 관리는 캐릭터 한 명당 한 주에 하나의 기록을 사용합니다.
- 동일 캐릭터와 동일 주차의 중복 기록을 허용하지 않습니다.
- 캐릭터의 관리 설정에 따라 주간 필수 항목을 계산합니다.
- 관리 대상이 아닌 항목은 미완료로 계산하지 않습니다.
- 보스 완료율과 전체 주간 완료율을 분리합니다.
- 결정석은 주간 관리에서 일괄 판매 상태를 관리합니다.
- 결정석 수익은 클리어 시 확정 수익으로 처리합니다.
- 목표 메소는 누적 수익이 아니라 실제 보유 메소를 기준으로 관리합니다.
- 확보 예정 메소와 실제 보유 메소를 분리합니다.
- Dashboard는 원본 데이터베이스가 아닌 조회 및 실행 화면으로 사용합니다.
- Dashboard 상단에는 미완료 항목과 빠른 입력 기능을 배치합니다.
- 기간별 수익은 Activity의 확정 수익을 기준으로 집계합니다.
- 유형별 통계를 위해 Activity에 보조 Formula 속성을 둡니다.
- Notion Formula의 한계가 있는 검증은 Python 자동화에서 처리합니다.
- 초기 Notion 구현은 마스터, Activity, 거래 상세, 요약, Dashboard 순서로 진행합니다.
- 테스트 데이터를 이용해 Relation, Rollup, Formula를 검증합니다.
- v0.2 초기 구현 완료 조건을 문서로 정의합니다.

# 45. 데이터베이스 설계 v0.2 최종 확정 사항

Maple ERP v0.2의 데이터베이스 구조는 다음과 같이 확정합니다.

```text
마스터 데이터
├── 캐릭터
├── 보스 마스터
└── 사냥터 마스터

활동 데이터
└── Activity

거래 상세 데이터
├── 보스 기록
├── 사냥 기록
└── 기타 수익 기록

요약 및 계획 데이터
├── 주간 관리
└── 목표 메소

표시 계층
└── Dashboard
```

핵심 데이터 흐름은 다음과 같습니다.

```text
캐릭터 및 마스터 데이터
        │
        ▼
Activity
        │
        ▼
거래 상세 기록
        │
        ▼
예상 수익 및 확정 수익
        │
        ▼
주간 관리 및 캐릭터 Rollup
        │
        ▼
Dashboard
```

Maple ERP는 다음 원칙을 유지합니다.

- Database First
- API First
- Single Source of Truth
- Activity 중심 설계
- 예상 수익과 확정 수익 분리
- 중복 입력 최소화
- 과거 기록 보존
- 자동화 우선
- 확장 가능한 구조
