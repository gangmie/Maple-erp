# Part 1. 공통 설계 원칙

## 1. 문서 목적

이 문서는 Maple ERP에서 사용하는 데이터베이스의 구조, 역할, 데이터 흐름, 명명 규칙을 정의합니다.

Maple ERP는 단순한 플레이 기록장이 아니라 다음 기능을 장기간 안정적으로 제공하는 개인 관리 시스템을 목표로 합니다.

- 캐릭터 정보 관리
- 플레이 활동 기록
- 보스 및 사냥 수익 계산
- 주간 진행 현황 관리
- 목표 메소 관리
- 메이플스토리 Open API 연동
- Notion API 자동 동기화
- GitHub Actions 기반 자동화
- 플레이 통계 및 Dashboard 제공

이 문서에서 정의한 데이터베이스 이름과 속성 이름은 이후 Notion API 코드에서도 사용합니다.

따라서 구현 이후에는 특별한 이유 없이 이름을 변경하지 않습니다.

---

# 2. 핵심 데이터 구조

Maple ERP의 데이터는 아래 네 개 계층으로 구분합니다.

1. 마스터 데이터
2. 활동 데이터
3. 거래 상세 데이터
4. 요약 데이터

```text
마스터 데이터
    │
    ├── 캐릭터
    ├── 보스 마스터
    └── 사냥터 마스터
          │
          ▼
활동 데이터
    │
    └── Activity
          │
          ├── 보스 기록
          ├── 사냥 기록
          └── 기타 수익 기록
                 │
                 ▼
요약 데이터
    │
    ├── 주간 관리
    └── 목표 메소
          │
          ▼
Dashboard
```

---

# 3. 각 데이터 계층의 역할

## 3.1 마스터 데이터

마스터 데이터는 여러 기록에서 반복해서 사용하는 기준 정보입니다.

예시:

- 캐릭터명
- 직업
- 월드
- 보스명
- 보스 난이도
- 결정석 기준 가격
- 사냥터명
- 지역
- 요구 포스

마스터 정보는 하나의 데이터베이스에서만 관리합니다.

보스 기록이나 사냥 기록에 동일한 정보를 반복해서 직접 입력하지 않고 Relation과 Rollup으로 가져옵니다.

---

## 3.2 활동 데이터

Activity는 사용자가 메이플스토리에서 수행한 모든 플레이 활동을 기록하는 공통 원장입니다.

예시:

- 보스 클리어
- 사냥
- 채집
- 제작
- 경매장 거래
- 몬스터파크
- 이벤트
- 무릉도장
- 기타 콘텐츠

Activity에는 모든 활동이 공통으로 가지는 정보만 저장합니다.

예시:

- 활동 날짜
- 캐릭터
- 활동 유형
- 소요 시간
- 완료 여부
- 주간 관리
- 메모

보스명, 사냥터, 결정석 수익처럼 특정 활동에만 필요한 정보는 Activity에 저장하지 않습니다.

---

## 3.3 거래 상세 데이터

거래 상세 데이터는 Activity에 연결되는 활동별 상세 기록입니다.

초기 구현 대상은 다음과 같습니다.

- 보스 기록
- 사냥 기록
- 기타 수익 기록

예를 들어 보스를 한 번 클리어했다면 다음 두 기록이 연결됩니다.

```text
Activity
- 날짜
- 캐릭터
- 활동 유형: 보스
- 소요 시간
- 완료 여부

보스 기록
- 보스
- 결정석 수익
- 드롭 아이템
- 예상 드롭 수익
- 확정 드롭 수익
```

Activity는 공통 정보의 원본이고, 보스 기록은 보스 전용 정보의 원본입니다.

---

## 3.4 요약 데이터

요약 데이터는 특정 기간이나 목표를 기준으로 활동 및 거래 데이터를 집계합니다.

예시:

- 캐릭터별 주간 보스 진행률
- 이번 주 결정석 수익
- 이번 주 사냥 수익
- 주간 총 플레이 시간
- 목표 메소 진행률

요약 데이터에는 가능하면 결과를 직접 입력하지 않습니다.

Relation, Rollup, Formula를 이용해 원본 데이터에서 계산합니다.

---

## 3.5 Dashboard

Dashboard는 데이터를 저장하는 데이터베이스가 아닙니다.

각 데이터베이스의 연결된 View와 집계 결과를 보여주는 화면입니다.

Dashboard에서는 원칙적으로 데이터를 직접 입력하지 않습니다.

다만 입력 편의를 위한 버튼이나 데이터베이스 템플릿은 배치할 수 있습니다.

예시:

- 사냥 기록 추가
- 보스 클리어 기록 추가
- 기타 수익 기록 추가
- 이번 주 관리 기록 생성

---

# 4. Activity 계층 설계

## 4.1 Activity의 목적

Activity는 모든 플레이 활동을 하나의 시간순 기록으로 관리하기 위한 공통 데이터베이스입니다.

Activity를 사용하면 다음 기능을 구현할 수 있습니다.

- 최근 활동 Timeline
- 오늘 수행한 활동 수
- 캐릭터별 플레이 시간
- 콘텐츠별 플레이 비중
- 주간 총 활동 수
- 월간 플레이 통계
- 마지막 플레이 날짜
- 활동 유형별 통합 검색

---

## 4.2 Activity와 상세 기록의 관계

Activity 한 건은 원칙적으로 하나의 상세 기록과 연결합니다.

```text
Activity 1건
    │
    ├── 보스 기록 0 또는 1건
    ├── 사냥 기록 0 또는 1건
    └── 기타 수익 기록 0 또는 1건
```

예시:

```text
Activity 유형이 "보스"인 경우
→ 보스 기록 1건과 연결

Activity 유형이 "사냥"인 경우
→ 사냥 기록 1건과 연결

Activity 유형이 "채집"인 경우
→ 기타 수익 기록 1건과 연결
```

하나의 Activity가 여러 종류의 상세 기록과 동시에 연결되지 않도록 합니다.

---

## 4.3 Activity가 필요한 이유

Activity 없이 보스 기록, 사냥 기록, 수익 기록만 각각 관리하면 다음과 같은 기능을 만들기 어렵습니다.

- 모든 활동을 시간순으로 한 번에 보기
- 전체 플레이 시간을 한 곳에서 계산하기
- 최근 플레이 활동을 통합해서 보여주기
- 콘텐츠 종류별 이용 비율 비교하기

Activity를 사용하면 상세 데이터베이스가 추가되더라도 공통 통계 구조를 변경할 필요가 없습니다.

예를 들어 추후 무릉도장 기록을 추가하더라도 다음처럼 확장할 수 있습니다.

```text
Activity
    ├── 보스 기록
    ├── 사냥 기록
    ├── 기타 수익 기록
    └── 무릉 기록
```

---

## 4.4 Activity의 한계와 구현 원칙

Notion Database에는 객체지향 프로그래밍의 상속 기능이 없습니다.

따라서 Activity와 상세 기록을 완전히 자동으로 하나의 객체처럼 처리할 수는 없습니다.

이를 보완하기 위해 다음 원칙을 적용합니다.

1. 공통 속성은 Activity에만 저장합니다.
2. 활동별 전용 속성은 상세 기록에만 저장합니다.
3. Activity와 상세 기록은 Relation으로 연결합니다.
4. 동일한 정보를 두 데이터베이스에 중복 저장하지 않습니다.
5. 템플릿 또는 버튼을 사용해 Activity와 상세 기록 생성을 간소화합니다.
6. 향후 Python 자동화로 두 기록의 생성을 동시에 처리할 수 있게 설계합니다.


# 6. 핵심 설계 원칙

## 6.1 Database First

모든 기능은 데이터베이스 구조를 먼저 설계한 뒤 구현합니다.

Dashboard나 버튼을 먼저 만들지 않습니다.

---

## 6.2 API First

메이플스토리 Open API에서 제공하는 데이터는 직접 입력하지 않는 것을 원칙으로 합니다.

API에서 제공하지 않는 정보만 사용자가 직접 입력합니다.

---

## 6.3 Single Source of Truth

하나의 정보는 하나의 원본 데이터베이스에서만 관리합니다.

예시:

- 캐릭터 직업의 원본: 캐릭터
- 결정석 기준 가격의 원본: 보스 마스터
- 보스 클리어 결과의 원본: 보스 기록
- 사냥 수익의 원본: 사냥 기록
- 활동 날짜와 소요 시간의 원본: Activity

다른 데이터베이스에서는 Relation, Rollup, Formula로 참조합니다.

---

## 6.4 Automation First

반복해서 입력하는 값은 템플릿, 버튼, Formula 또는 Python 자동화로 처리합니다.

자동화 후보:

- Activity 기록명 생성
- 주차 시작일 계산
- 주간 관리 기록 연결
- 보스 기록과 Activity 동시 생성
- 사냥 기록과 Activity 동시 생성
- 매주 새로운 주간 관리 기록 생성
- API 캐릭터 정보 갱신

---

## 6.5 데이터 누적 원칙

거래 데이터와 활동 데이터는 삭제보다 누적을 기본으로 합니다.

과거 기록을 수정해야 할 경우에도 가능한 한 기록을 삭제하지 않고 값을 수정합니다.

게임 업데이트로 기준값이 변경되어도 과거 기록은 당시 값을 유지해야 합니다.

---

## 6.6 확장성

다음 변화가 발생해도 기존 구조를 다시 만들지 않는 것을 목표로 합니다.

- 캐릭터 수 증가
- 새로운 월드 추가
- 새로운 보스 추가
- 새로운 사냥터 추가
- 새로운 콘텐츠 추가
- 다년간 데이터 누적
- Web Dashboard 추가
- Discord 알림 추가

# 8. 명명 규칙

## 8.1 문서 언어

- GitHub 문서: 한국어
- Notion 데이터베이스명: 한국어
- Notion 속성명: 한국어
- Python 파일명: 영어
- Python 변수명 및 함수명: 영어
- Git Commit Message: 영어

---

## 8.2 데이터베이스 이름

데이터베이스 이름은 짧고 명확한 명사형으로 작성합니다.

사용 예시:

- 캐릭터
- 보스 마스터
- Activity
- 보스 기록
- 사냥 기록
- 주간 관리

사용하지 않는 예시:

- 캐릭터를 관리하는 데이터베이스
- 이번 주 보스 기록 목록
- 사냥해서 번 메소 기록

---

## 8.3 Notion 속성 이름

속성 이름은 사용자가 읽기 쉬운 한국어로 작성합니다.

가능하면 의미가 명확한 이름을 사용합니다.

예시:

- 최근 API 갱신 시간
- 예상 드롭 수익
- 확정 드롭 수익
- 결정석 일괄 판매 완료
- 활동 유형

모호한 축약어는 사용하지 않습니다.

---

## 8.4 날짜 속성

날짜 속성은 용도를 명확히 표시합니다.

예시:

- 활동 날짜
- 클리어 날짜
- 판매일
- 주차 시작일
- 주차 종료일
- 최근 API 갱신 시간

---

## 8.5 숫자 속성

숫자 속성에는 가능한 한 단위를 이름에 포함합니다.

예시:

- 소요 시간(분)
- 사냥 시간(분)
- 경험치 증가율(%)

메소 단위 숫자는 기본적으로 실제 메소 단위의 정수로 저장합니다.

예시:

```text
1억 메소 → 100000000
```

표시 형식은 Notion에서 천 단위 구분기호를 사용합니다.

---

## 8.6 체크박스 속성

예시:

- 클리어 당시 결정석 금액
- 활동 당시 캐릭터 레벨
- 판매 당시 아이템 가격

스냅샷 속성은 자동화 단계에서 필요성을 검토합니다.

## 14.2 목적

`캐릭터` 데이터베이스는 Maple ERP에서 관리하는 모든 캐릭터의 기준 정보와 운영 설정을 저장합니다.

단순히 레벨과 직업을 기록하는 데이터베이스가 아니라, 각 캐릭터를 Maple ERP에서 어떤 방식으로 관리할지 결정하는 설정 데이터베이스 역할도 담당합니다.

캐릭터별로 다음 내용을 제어할 수 있어야 합니다.

- API 자동 갱신 여부
- 본캐 및 보스돌이 구분
- 보스 관리 여부
- 사냥 관리 여부
- 수로 및 플래그 관리 여부
- 심볼 및 HEXA 관리 여부
- Dashboard 표시 여부
- 알림 및 자동화 대상 여부
- 활성, 육성 중, 휴면 등의 운영 상태

## 14.4 식별자 설계

캐릭터명은 사용자가 확인하기 쉬운 표시용 이름입니다.

캐릭터명은 변경될 수 있으므로 내부 식별자로 사용하지 않습니다.

Maple ERP에서는 다음 두 식별자를 구분합니다.

### 캐릭터 ID

Maple ERP 내부에서 사용하는 고유 식별자입니다.

예시:

```text
CHR-0001
CHR-0002
CHR-0003
```

캐릭터명이 변경되어도 캐릭터 ID는 변경하지 않습니다.

### OCID

메이플스토리 Open API에서 캐릭터 정보를 조회하기 위해 사용하는 외부 식별자입니다.

OCID는 API 동기화 과정에서 관리합니다.

OCID가 변경되거나 다시 조회되어야 하는 상황에 대비하여 캐릭터 ID와 별도로 저장합니다.

### 경험치 속성을 텍스트로 저장하는 이유

매우 큰 정수는 시스템에 따라 숫자 정밀도 문제가 발생할 수 있습니다.

따라서 정확한 원본 경험치는 텍스트로 보관하고, Dashboard에서 활용하기 쉬운 경험치율은 별도 숫자 속성으로 저장합니다.

### 유니온 레벨 주의사항

유니온 정보는 캐릭터 자체만의 고유 능력치라기보다 계정과 월드 기준 정보에 가깝습니다.

초기 버전에서는 API 조회 편의를 위해 캐릭터에 저장할 수 있습니다.

향후 여러 캐릭터에서 동일한 유니온 값이 반복 저장되는 문제가 커지면 `계정·월드 프로필` 마스터로 분리합니다.

## 14.10 자동화 설정 속성

| 자동 주간 생성 | 체크박스 | 해제 | 사용자 | 매주 주간 관리 기록 자동 생성 여부 |
| 알림 사용 여부 | 체크박스 | 해제 | 사용자 | Discord 등 외부 알림 대상 여부 |

### 동기화 상태 옵션

- 대기
- 진행 중
- 성공
- 실패
- 비활성

### API 사용 여부와 자동 갱신 여부의 차이

`API 사용 여부`는 해당 캐릭터가 API 정보 관리 대상인지를 의미합니다.

`자동 갱신 여부`는 GitHub Actions가 정해진 시간에 자동으로 조회할지를 의미합니다.

예를 들어 API 사용은 하지만 자동 갱신을 해제하고 수동 실행만 허용할 수 있습니다.

### 조건부 Rollup 주의사항

Notion Rollup만으로 활동 유형별 조건부 합계가 제한될 수 있습니다.

필요하다면 Activity에 다음 Formula 속성을 둡니다.

- 보스 확정 수익
- 사냥 확정 수익
- 기타 확정 수익

예시:

```text
활동 유형이 보스이면 확정 수익
그렇지 않으면 0
```

캐릭터에서는 해당 Formula 속성을 Rollup으로 합산합니다.

## 14.14 Formula 속성

### 관리 상태

캐릭터의 운영 상태를 한눈에 표시합니다.

개념식:

```text
현재 관리 대상이 해제되어 있으면 "관리 제외"

캐릭터 상태가 휴면 또는 은퇴이면 해당 상태 표시

그 외에는 "관리 중"
```

### API 상태 경고

개념식:

```text
API 사용 여부가 활성이고 동기화 상태가 실패이면 "확인 필요"

API 사용 여부가 활성이고 최근 갱신이 오래되었으면 "갱신 지연"

그 외에는 "정상"
```

정확한 Formula 문법과 갱신 지연 기준은 Notion 구현 단계에서 확정합니다.

### 관리 기능 수

활성화된 기능별 관리 체크박스 개수를 합산합니다.

Dashboard에서 관리 복잡도를 확인하거나 설정 누락을 점검하는 데 사용할 수 있습니다.

## 14.16 자동화 규칙

### 신규 캐릭터 등록

1. 사용자가 캐릭터명을 입력합니다.
2. 자동화가 캐릭터 ID를 생성합니다.
3. 캐릭터명으로 OCID를 조회합니다.
4. 기본 정보를 가져옵니다.
5. API 속성을 갱신합니다.
6. 동기화 상태를 성공으로 변경합니다.
7. 사용자가 관리 설정을 확인합니다.

### 정기 동기화

자동화 대상 조건:

```text
현재 관리 대상 = 체크
API 사용 여부 = 체크
자동 갱신 여부 = 체크
캐릭터 상태가 휴면, 은퇴, 삭제 예정이 아님
```

처리 순서:

1. OCID 유효성 확인
2. 캐릭터 기본 정보 조회
3. 필요한 상세 API 조회
4. Notion 속성 갱신
5. 최근 API 갱신 시간 기록
6. 동기화 성공 또는 실패 상태 기록

### 닉네임 변경 대응

캐릭터 ID는 유지합니다.

API 조회 결과에서 캐릭터명이 변경된 경우 `캐릭터명`만 최신 이름으로 갱신합니다.

OCID 재조회가 필요한 경우 캐릭터명을 기준으로 다시 조회하고, 변경된 OCID를 저장합니다.

## 14.19 데이터 무결성 규칙

1. 캐릭터 ID는 중복될 수 없습니다.
2. 캐릭터 ID는 생성 후 변경하지 않습니다.
3. 캐릭터명 변경 시 기존 캐릭터 행을 새로 만들지 않습니다.
4. 캐릭터명이 변경되어도 기존 Activity Relation을 유지합니다.
5. API가 갱신하는 속성과 사용자가 입력하는 설정 속성을 구분합니다.
6. API 자동화는 사용자 설정 속성을 덮어쓰지 않습니다.
7. 거래 기록이 존재하는 캐릭터는 가능한 한 삭제하지 않습니다.
8. 사용하지 않는 캐릭터는 휴면 또는 은퇴 상태로 전환합니다.
9. OCID만으로 동일 캐릭터 여부를 영구적으로 판단하지 않습니다.
10. 캐릭터 상태가 변경되어도 과거 기록은 유지합니다.

## 15.2 목적

`보스 마스터`는 보스명과 난이도별 기준 정보를 한 곳에서 관리합니다.

보스 기록을 작성할 때 보스 이름, 난이도, 기준 결정석 가격 등을 매번 직접 입력하지 않고 Relation으로 선택할 수 있게 합니다.

보스명만으로는 난이도를 구분할 수 없으므로 한 행은 `보스명 + 난이도` 조합을 나타냅니다.

예시:

```text
스우 / Normal
스우 / Hard
스우 / Extreme
루시드 / Normal
루시드 / Hard
```

```text
BSM-0002
BSM-0003
```

### 보스 식별명

Notion 제목 속성입니다.

형식:

```text
보스명 / 난이도
```

예시:

```text
스우 / Hard
윌 / Normal
검은 마법사 / Hard
```

## 15.5 결정석 가격 관리

`결정석 기준 가격`에는 현재 기준 가격을 저장합니다.

실제 보스 클리어 기록에는 클리어 당시 결정석 가격을 별도 스냅샷으로 저장합니다.

따라서 보스 마스터의 가격을 변경해도 과거 보스 기록의 확정 수익은 바뀌지 않아야 합니다.

```text
보스 마스터
현재 결정석 기준 가격

↓

보스 기록 생성 시 복사

↓

실제 결정석 수익
```

결정석 가격 변경 이력이 필요해지면 향후 `결정석 가격 이력` 데이터베이스를 별도로 추가합니다.

- 보스 ID 자동 생성

- 보스 기록 생성 시 결정석 기준 가격 복사
- 보스 기록 생성 시 기본 예상 시간 복사
- 사용하지 않는 보스의 Dashboard 제외
- 결정석 가격 변경일 기록

---

## 15.9 데이터 무결성 규칙

1. `보스명 + 난이도` 조합은 중복될 수 없습니다.
2. 보스 ID는 생성 후 변경하지 않습니다.
3. 결정석 가격 변경이 과거 기록에 반영되지 않도록 합니다.
4. 삭제된 콘텐츠도 과거 기록이 있으면 보스 마스터 행을 삭제하지 않습니다.
5. 사용하지 않는 보스는 `사용 여부`를 해제합니다.
6. 보스 기록에서는 반드시 보스 마스터를 Relation으로 선택합니다.
7. 보스별 결정석 판매 체크박스는 사용하지 않습니다.
8. 결정석 판매 완료 여부는 주간 관리에서 일괄 관리합니다.

## 16.5 최소 권장 레벨의 의미

`최소 권장 레벨`은 공식 고정값이 아니라 사용자가 실제 사냥 경험을 기준으로 정하는 값입니다.

직업, 장비, 전투력, 사냥 빌드에 따라 효율이 다를 수 있으므로 객관적인 입장 조건과 구분합니다.

`몬스터 레벨`은 사냥터 기준 정보이고, `최소 권장 레벨`은 개인 운영 정보입니다.

이 구조는 새로운 포스 체계가 추가되어도 속성을 다시 만들 필요가 없다는 장점이 있습니다.

- 사냥 기록 생성 시 사냥터 기준 정보 연결

## 16.10 데이터 무결성 규칙

1. 사냥터 ID는 중복될 수 없습니다.
2. 사냥터명 변경 시 기존 행을 새로 만들지 않습니다.
3. 사냥 기록은 반드시 사냥터 마스터와 연결합니다.
4. 현재 사용하지 않는 사냥터도 과거 기록이 있으면 삭제하지 않습니다.
5. 사용하지 않는 사냥터는 `사용 여부`를 해제합니다.
6. 사용자 평가와 객관적 기준 정보를 구분합니다.
7. 몬스터 레벨과 최소 권장 레벨을 같은 의미로 사용하지 않습니다.
8. 요구 포스는 포스 종류와 함께 입력합니다.
9. 게임 업데이트로 맵이 변경되면 최근 정보 확인일을 갱신합니다.

# 17. 마스터 데이터 간 관계

```text
캐릭터
    │
    ├── Activity
    ├── 주간 관리
    └── 추천 사냥터
          │
          ▼
사냥터 마스터
    │
    └── 사냥 기록

보스 마스터
    │
    └── 보스 기록
```

초기 버전에서는 캐릭터, 보스 마스터, 사냥터 마스터가 서로 직접 강하게 의존하지 않도록 설계합니다.

실제 플레이 기록은 Activity와 거래 상세 데이터베이스를 통해 연결합니다.

---

# 18. 마스터 데이터 API 적용 범위

| 데이터베이스 | API 자동 갱신 | 사용자 직접 관리 |
|---|---|---|
| 캐릭터 | 기본 정보, 능력치, 심볼 등 | 운영 설정 및 기능 설정 |
| 보스 마스터 | 초기 버전에서는 없음 | 보스, 난이도, 결정석 기준 가격 |
| 사냥터 마스터 | 초기 버전에서는 없음 | 지역, 몬스터, 포스 및 개인 평가 |

API가 제공하는 캐릭터 데이터와 사용자가 설정하는 ERP 운영 속성을 명확히 구분합니다.

자동화는 사용자가 직접 설정한 관리 체크박스와 상태값을 수정하지 않습니다.

---

# 19. Part 2 확정 사항

이번 Part에서 확정한 사항은 다음과 같습니다.

- 캐릭터 마스터를 정보 저장소이자 ERP 설정 데이터베이스로 사용합니다.
- 캐릭터명과 OCID를 영구 내부 기본키로 사용하지 않습니다.
- Maple ERP 내부의 고정된 `캐릭터 ID`를 별도로 사용합니다.
- OCID는 Open API 조회를 위한 외부 식별자로 관리합니다.
- 캐릭터별 기능 관리 여부를 체크박스로 설정합니다.
- 캐릭터 상태를 이용해 과거 기록을 삭제하지 않고 운영 대상에서 제외합니다.
- API 정보와 사용자 설정 정보를 명확하게 분리합니다.
- 보스 마스터는 보스명과 난이도 조합을 한 행으로 관리합니다.
- 보스 마스터의 현재 결정석 가격과 과거 보스 기록의 실제 가격을 분리합니다.
- 결정석 판매 여부는 개별 보스 기록이 아니라 주간 관리에서 일괄 관리합니다.
- 사냥터 마스터는 객관적 정보와 사용자 개인 평가를 함께 관리합니다.
- 아케인포스와 어센틱포스 요구량을 `포스 종류 + 요구 포스` 구조로 통합합니다.
- 마스터 데이터는 삭제보다 비활성화를 기본으로 합니다.
- API가 제공하지 않는 보스 및 사냥터 기준 정보는 사용자가 직접 관리합니다.
